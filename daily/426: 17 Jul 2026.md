# Log 426: At Finalization

> 17 July 2026

## What I worked on

- [Continued] the async static invoice refresh branch by narrowing where
  direct refreshes belong and where batching still helps.
- Moved the channel-close refresh into `finish_close_channel`, so close
  finalization owns the static invoice update it can make stale.
- Kept the pending refresh bit for peer `ChannelUpdate`s, where inline refresh
  would turn forwarding churn into repeated static invoice updates.
- Revisited whether splices should force-refresh static invoices and traced the
  current async receive cache back to amountless reusable offers.
- Separated the current LDK async receive cache from the broader
  `StaticInvoice` abstraction while checking the currency branch's
  `InsufficientAmount` path.

## Reflection

The static invoice refresh work became more precise today because the question
was no longer whether stale cached invoices matter. They do. The harder part
was deciding which channel changes deserve an immediate refresh and which ones
should keep using the delayed path.

The old pending flag had a reasonable shape when viewed from a distance. It
gave `ChannelManager` a way to say that channel state had changed and cached
async receive static invoices might need to be rebuilt later. That protected
the refresh from running while channel state was still moving, and it also
coalesced repeated peer `ChannelUpdate`s into one timer-driven update.

Review made that broad answer feel too blunt for channel closes. A closed
channel may have appeared in static invoices served on our behalf, and
`finish_close_channel` is the point where the close has actually been
finalized. It already runs outside the peer locks that made inline refreshes
awkward elsewhere. Moving `force_refresh_async_receive_static_invoices()` into
that function made the ownership match the event: once the channel is
finalized, the refresh belongs there.

Peer forwarding updates still had a different pressure. Refreshing every time
a peer sends a `ChannelUpdate` would replace one stale-invoice concern with a
burst of static invoice updates. Keeping the pending bit there kept the useful
part of the old design: the cache can be marked stale immediately without
rebuilding and sending every intermediate version.

The splice question forced the amount model into the foreground. A splice-out
can reduce inbound capacity, so it is easy to imagine it making a route hint
less useful for an amountful receive flow. Reading the async receive builder
path showed that the current cached offers are amountless:
`get_async_receive_offer()` has no amount parameter,
`create_async_receive_offer_builder` does not set one, and
`create_static_invoice_for_server` passes `None`. In that model, splice
capacity affects route-hint quality more than invoice correctness.

That also made the offline-merchant shape easier to see. The current cache is
a bounded pool of prebuilt reusable offers and static invoices, rotated through
server slots and refreshed as they are used. It fits amountless async receive
well. Amountful offline commerce would need a different cache and API shape,
because price would have to enter before the static invoice is built instead
of after a reusable pool entry is already waiting.

The currency branch question at the end landed on the same boundary. Today's
LDK-generated async receive offers may not reach the `InsufficientAmount`
branch because they are amountless, but `StaticInvoice` can still represent a
currency-denominated offer built outside that cache path. The payer-side check
is still doing real work for that broader case: before sending the payment, it
confirms that the original invoice request amount still satisfies the offer.

By the end, the refresh logic felt less like one shared fallback and more like
several decisions tied to their actual source of staleness. Channel close
finalization refreshes directly. Peer update churn stays batched. Splices stay
out of the forced-refresh set for the current amountless cache model. The
remaining amountful story belongs to a larger API question, not a hidden
condition inside this branch.
