# Log 387: One Place to Stand

> 20 May 2026

## What I worked on

- [Continued] the BOLT 12 currency work on the `currency-98` branch by
  threading `CurrencyConversion` through `ChannelManager` setup, read args,
  helper aliases, fuzz scaffolding, async payment paths, and deserialization so
  offer flows can carry one conversion dependency end to end.
- Expanded `OfferBuilder` so currency-denominated amounts can be constructed
  through the public API instead of existing only as representable internal
  state.
- Moved build-time validation onto converted currency ranges and made it reject
  zero, misordered, and over-cap results so fiat pricing cannot slip around the
  same limits already enforced for direct bitcoin amounts.
- Carried the new amount handling through offer serving and payment-side paths,
  including the switch to the newer payable-amount semantics where the request
  amount and the derived amount needed to stay distinct.
- Reworked the surrounding tests so older fixed-amount assumptions no longer
  break the flow, then added focused coverage for valid currency offers and the
  zero-msat converted-minimum failure case.

## Reflection

Today was mostly plumbing, but the kind that decides whether a feature is real
or still just posing at the edges.

The amount model had already started making room for currency-denominated
offers. What it still did not have was one route through the stack that could
carry that decision without quietly dropping back into millisatoshi-only
assumptions. `ChannelManager`, builder validation, serving paths, and payment
handling all had to agree on where conversion lived, otherwise currency support
would only be true in whichever layer I had touched most recently.

That is what made the validation work feel important instead of procedural. It
was not enough to let the builder accept `Amount::Currency`. The converted
result still had to land inside the same safety envelope as any direct bitcoin
amount, or the new feature would have created a softer path around limits the
rest of the code already relied on.

By the end of the day, the change I trusted most was not the public builder
surface. It was that the conversion dependency had one place to stand. The
feature stopped looking like an isolated type improvement and started reading
like something the normal offer flow could actually carry.
