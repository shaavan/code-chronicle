# Log 415: Who Chooses Ten

> 30 June 2026

## What I worked on

- [Started] the static-invoice server configuration branch by confirming that
  recipients still allocate ten offer slots regardless of how many invoices a
  server application is willing to persist.
- Traced the numbered-slot lifecycle from the recipient's offer-path request
  through static invoice construction, LDK's persistence event, and the
  application's database write.
- Made the server's per-recipient allowance part of the authenticated blinded
  path context, then returned it to the recipient in the `OfferPaths` response.
- Added optional TLV fields for the allowance while preserving ten as the
  meaning of messages and contexts that predate the field.
- Introduced an opt-in API for configuring a non-zero limit and kept the
  existing server API on its legacy behavior.

## Reflection

The fixed limit of ten had looked like a cache detail until I followed each
slot across the server boundary. The recipient creates the offers and static
invoices, while the application behind the server owns the database that must
hold them. A persistence rejection can say that storage failed, but it arrives
too late to give both sides the same idea of how large the cache should be.

The slot number helped locate where that agreement belonged. It already ties a
recipient-side offer to the invoice the server stores and later refreshes. If
the server chooses the range of valid slots when it creates the onboarding
paths, the allowance can travel through the same exchange without becoming a
recipient-controlled claim.

Putting the value in the blinded-path context gave the server something it can
authenticate when the request returns. Sending it back in `OfferPaths` then
gives the recipient enough information to size its cache. The round trip now
has one authority for the limit and one value both sides can act on.

Compatibility kept the first part of the implementation deliberately quiet.
The new fields are optional, use fresh TLV numbers, and leave an absent value
meaning ten. The configuration API follows the same shape: servers can opt in
to a non-zero allowance, while existing callers continue unchanged. These
commits establish who chooses the number and how it crosses the protocol. The
next part can enforce it without having to reopen that boundary.
