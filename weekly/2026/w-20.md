# Weekly Log 20: Only What the Stack Knows

> 13 May 2026 – 18 May 2026

## Overview

This week was mostly about taking recurrence support the rest of the way from
promising structure into something the stack could actually carry without
guessing.

The early part of the week stayed inside the same recurrence thread, but the
work kept getting more specific. I moved token verification and basetime
resolution into the response flow that already owns the full request context,
then tightened the request model so basetime and non-basetime recurrence stop
blurring into one loose set of optional fields. By the time I opened the first
recurrence PR, the useful change was not just that more code compiled. Offer,
`InvoiceRequest`, and invoice recurrence were finally describing the same thing
through one stack instead of three nearby approximations. On 18 May I switched
back to currency work, but it turned out to be the same kind of correction:
stop asking a type to imply state it does not actually hold.

## Highlights

### Moving recurrence decisions to the place that can defend them

- Shifted recurrence-token verification and basetime resolution out of invoice
  construction and into the offer response flow, so follow-up recurrence is
  admitted from full request context instead of builder-side fallback logic.
  ([Branch][Recurrence])

- Simplified the continuation-token shape in the same pass, cutting redundant
  `offer_id` handling and making next-token emission come directly from
  resolved recurrence state rather than invoice-local reconstruction.

- Carried the resolved basetime through the respond paths into invoice
  creation, which made cancellations, follow-up requests, and token minting
  read like one checked stream instead of a set of partially coordinated
  helpers.

### Making the request and invoice models say one clear thing

- Replaced the older scattered recurrence fields on `InvoiceRequest` with one
  explicit model for basetime-anchored requests and streams that begin only
  after the first successful payment.

- Tightened validation so offers with an explicit recurrence basetime must use
  the basetime-anchored request form, while basetime-less offers stay on the
  non-basetime path and initial requests cannot smuggle in follow-up-only
  fields.

- Finished the broader recurrence refactor by bringing offer, request, and
  invoice recurrence onto the same structured model, then cleaned up parsing,
  serialization, docs, and tests until the first recurrence [PR] could stand
  as one reviewable unit.

### Carrying the same discipline into currency amounts

- Returned to the `currency-98` branch and split the amount explicitly named
  in an `InvoiceRequest` from the payable amount that only exists after the
  offer, quantity, and conversion rules are applied. ([Branch][Currency])

- Reworked request-side amount helpers so downstream logic stops inferring
  intent through shape tests and instead resolves missing, unsupported, and
  overflow cases through one payable-amount path.

- Removed the extra `MsatsRange` staging layer and moved accepted
  exchange-rate bounds closer to `Amount`, so the model stays nearer to where
  currency values are actually consumed.

## Reflections

The week kept improving each time I stopped letting one layer compensate for
context owned somewhere else.

That was the real pressure inside the recurrence work. The builder fallback was
too willing to reconstruct meaning after the important checks had already gone
by. The request model was still carrying combinations that looked flexible only
because they had not been forced to name which kind of recurrence they were.
The invoice side still had remnants of its own local version of the same idea.
Each fix came from holding those boundaries still a little longer. Once the
response flow owned recurrence admission, once the request had to declare which
form it was using, and once the invoice side stopped keeping its own
approximation, the branch got narrower but much easier to trust.

Opening the first recurrence PR mattered for that reason. It felt less like
packaging a long-running branch and more like reaching a point where the stack
could finally describe one coherent model from offer through invoice without
quietly inventing pieces along the way.

The switch back to currency at the end of the week did not really break that
arc. It showed the same weakness in a different costume. The amount model was
still letting request data and later derived value blur together, which pushed
too much interpretation into downstream callers. Splitting those apart made the
currency work feel more direct immediately. By the end of the week, both
threads were asking for the same standard: types and layers should name only
what they actually know, and the rest of the flow should get simpler once they
do.
