## Reflection

Today had a small handoff in it. The dummy-hop branch had been absorbing most
of my attention because every failure still seemed to say something about the
shape of the feature. Once the CI pass was done and the PR update was written,
that work finally had a reviewable edge again. The fee bucketing and dummy TLV
selection now sit together instead of being spread across the history of the
branch, which makes the argument easier to follow.

That also made the next piece of work feel less like context switching and more
like picking up the next loose boundary. The async-payment issue starts from a
different kind of tension. A static invoice wants to be a stable object, but
the channel state behind it keeps moving. If the payment path depends on data
that can age underneath it, then the useful question is where fresh state is
allowed to reenter without turning the whole flow into something too mutable.

Most of the day was spent finding that boundary rather than changing it. I
looked at where the invoice data is collected, where live channel state still
matters, and what kind of update point could keep the payment description
honest. It is early work, but the shape is already concrete enough: the invoice
can stay static as an interface only if the path has a deliberate way to refresh
the channel-dependent parts that decide what can actually be paid.
