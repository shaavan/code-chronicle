# Log 379: What the Token Carries

> 08 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  defining the continuation-token primitive itself instead of extending the
  surrounding recurrence field plumbing any further first.
- Replaced the earlier plaintext basetime-token idea with an authenticated
  token shaped as `encrypted_basetime || nonce || hmac`, so the payee can
  recover recurrence basetime without extra server-side state while the token
  stays bound to the offer, payer identity, and expected recurrence context.
- Kept the implementation layered by adding the low-level token creation and
  verification helpers in `offers::signer`, extending `ExpandedKey` with
  generic offer-bound encryption support, and keeping recurrence-specific
  wrappers at the `invoice_request.rs` layer instead of pushing that behavior
  into `flow.rs` too early.
- Added focused coverage for token creation, round-trip verification, and
  tamper rejection through the verified `InvoiceRequest` wrappers, then cleaned
  up the surrounding docs and commit message so the feature scope reads as
  helper plumbing rather than invoice-generation behavior.
- Verified the path with formatting, compile-oriented checks, and targeted
  local cargo tests.

## Reflection

Today became more concrete once I stopped treating the token as a small field
addition and asked what it actually had to prove.

The plaintext basetime version only looked plausible as long as I kept the
question narrow. As soon as I worked through how a follow-up recurring payment
would rely on it, that shape stopped holding. Recovering one timestamp was not
enough. The token had to carry recurrence state in a way the payee could trust
later without inventing extra local storage or loosening what the stream stayed
bound to.

That pushed the work into a better boundary. The crypto helper needed to stay
small and honest, while the message layer kept ownership of what the token
means for recurring `InvoiceRequest`s specifically. I liked that split because
it kept the day from turning into premature flow logic. The implementation now
has a real token primitive under it, but the recurrence decisions that will
eventually depend on that primitive are still waiting in the layer that can
actually justify them.
