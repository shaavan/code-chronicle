# Weekly Log 18: Only Where It Holds

> 30 April 2026 – 05 May 2026

## Overview

This week stayed inside recurrence, but the work kept changing what counted as
real progress.

I started by auditing the branch for places where parsing, builder checks, and
internal state were still saying slightly different things. That led into a
fuller request-side recurrence model, then into speculative continuation-token
support on both invoice requests and invoices. The useful part was not adding
more recurrence surface for its own sake. It was making the branch stop
claiming support in places where the rules, data flow, or protocol status were
still too loose to defend.

## Highlights

### Making the request path say one thing

- Audited the `InvoiceRequest` recurrence stack after earlier cherry-picks and
  fixed stale validation, fixtures, and semantic drift so the branch was green
  again for the right reasons. ([Branch][Recurrence])

- Consolidated recurrence validation into one semantic check path so builder
  and parser behavior reject the same invalid combinations instead of
  disagreeing quietly.

- Corrected `RecurrenceLimit` handling to treat the value as a count rather
  than a zero-based maximum, and tightened the surrounding docs so the code and
  explanation now describe the same rule.

### Carrying the full recurrence state through requests

- Grouped `counter`, `start`, and `cancel` under a dedicated
  `InvoiceRequestRecurrence` shape and threaded it through request storage,
  builders, parsing, and serialization.

- Kept wire compatibility intact by preserving the older internal `counter` TLV
  and introducing separate optional TLVs for the newer recurrence fields
  instead of rewriting past encoding decisions.

- Left `recurrence_paywindow` explicitly deferred, but narrowed that gap to one
  concrete boundary: checks that need trusted current time or prior accepted
  recurrence state that this layer still does not own.

### Adding continuation tokens without pretending they are settled

- Added speculative continuation-token support to follow-up recurring
  `InvoiceRequest`s through a provisional odd TLV, with validation and tests
  that make clear the field belongs only to the continuation path.

- Reconciled the invoice-side recurrence stack with the current draft before
  extending it further, fixing basetime handling, explicit recurring-offer
  requirements, and refund-side semantic error mapping.

- Implemented the invoice-side half of the continuation token with the same
  constraints: recurring offer invoices can carry it, but empty values,
  non-recurring offers, refunds, and static invoices cannot.

## Reflections

The week kept stripping away versions of recurrence support that were easy to
live with only because I was already carrying the missing context in my head.

The first pass through `InvoiceRequest` exposed that kind of softness most
clearly. Some rules were enforced while parsing but not while building. Some
data was conceptually one thing but still traveled through the code in pieces.
Those are not dramatic failures, but they make the branch feel more finished
than it is. Pulling the checks together and regrouping the request-side
recurrence state made the surface harder to misread.

The continuation-token work pushed on the same problem from another side. The
idea is still speculative, so the implementation had to carry it without
letting repetition turn it into an implied standard. That meant keeping the
field provisional in both request and invoice paths, and being strict about
where it could exist at all. The invoice-side cleanup on 4 May ended up
mattering for the same reason. Before adding the second half of the token
flow, I needed the existing recurrence support to stand on its own terms
again.

By the end of the week, the branch felt narrower, but more usable. It now says
more plainly which recurrence behavior belongs to the current draft, which
parts are still experimental, and where this layer really does not have enough
context to claim more than it can prove.
