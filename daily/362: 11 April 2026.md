# Log 362: Closing the Gap

> 11 April 2026

## What I worked on

- [Continued] the currency conversion work.
- Finished reshaping the BOLT12 fiat-offer flow around the intended "omit
  now, verify later" model.
- Removed eager fiat conversion from invoice-request creation so omitted request
  amounts stay omitted on the wire until the payee constructs the invoice.
- Tightened payer-side validation so returned invoices are checked against the
  locally expected amount range rather than exact independently resolved values.
- Closed the remaining failure-handling gap by treating bad or unverifiable
  fiat invoices as terminal local failures and clearing pending payment state
  immediately.
- Cleaned up the supporting state and test coverage so the revised PR tells the
  design story more clearly.

## Reflection

Today was about closing the gap between the protocol model and the error model.

The amount-handling side finally feels aligned with the intended BOLT12 flow.
Earlier versions still pulled conversion work too far forward, which made the
request-building path carry responsibilities that really belong later when the
invoice exists and can actually be checked. Moving back to "omit now, verify
later" fixed more than the wire semantics. It made the payer and payee roles
feel cleaner again. Omitted request amounts stay omitted, the payee resolves
fiat-priced offers when constructing the invoice, and the payer validates the
result against the range it was already expecting instead of pretending two
independent conversions should match exactly.

That would have been a satisfying stopping point on its own, but the more
important part of the day was finishing the failure story around it. Once the
normal path became more coherent, the remaining weakness stood out more
plainly: bad or unverifiable fiat invoices were still allowed to linger until
expiry even when the payer already had enough information to reject them. That
behavior was technically survivable, but it was the wrong shape. Failing them
immediately and clearing the pending state makes the system feel more honest
about what it knows and when it knows it.

By the end, the work had shifted from patching specific bugs to tightening the
contract of the whole flow. The feature now reads less like a set of exception
paths and more like a design that knows where amount resolution belongs, where
verification belongs, and where failure should become final.
