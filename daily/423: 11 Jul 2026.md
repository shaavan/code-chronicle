# Log 423: Missing Edges

> 11 July 2026

## What I worked on

- [Continued] the currency-payment branch by filling reviewer-noted coverage
  gaps around unsupported currencies, converted amount bounds, and invoice
  response validation.
- Added `pay_for_offer` coverage for offers whose currency cannot be converted
  before an invoice request is created and signed.
- Pinned the lower-bound arithmetic for `payable_amount_msats()` with an
  explicit quantity and tolerance boundary.
- Reworked the invoice-request deserialization fuzzer to use input-driven
  exchange rates and tolerances instead of stopping at
  `NullCurrencyConversion`.
- Covered the exact-boundary invoice response case and the static-invoice
  `InsufficientAmount` failure path.

## Reflection

Yesterday closed the arithmetic around positive converted amounts. Today moved
through the places where that arithmetic still needed witnesses.

The useful constraint was keeping each new test near the commit that owned the
behavior. It would have been easy to add one broad coverage commit at the end
of the branch and call the reviewer gaps handled. That would have made the
stack easier to patch in the moment, but harder to review later because the
test would no longer explain which invariant it was protecting. Mapping each
gap back to its commit made the branch feel more honest about where the
behavior changed.

The unsupported-currency payer case belonged at invoice-request creation. A
normal USD-only node should not create and sign a request for an EUR offer it
cannot price, even if another test converter can quote that currency. That
test is less about the specific currencies and more about the point of refusal:
failure needs to happen before a signed request leaves the payer.

The amount-bound tests had a smaller shape, but they mattered because the
branch depends on exact lower edges. A USD 10 offer, quantity two, and ten
percent lower tolerance gives an 18,000 msat minimum. Checking 17,999 against
18,000 makes the rule visible in a way a looser success or failure test would
not. The invoice response path needed the other side of that same line: a
request exactly at the converted minimum must still be accepted, signed, and
verifiable.

The fuzzer change was a different kind of coverage. `invoice_request_deser`
was using `NullCurrencyConversion`, which meant many currency requests stopped
at unsupported-currency handling before they reached the conversion math. The
new converter pulls rates and tolerances from fuzz input, so the target can
reach the arithmetic often enough to find real edges without pretending the
rates are stable or hand-picked.

The static-invoice `InsufficientAmount` branch tied the coverage back to the
payment state. A matching invoice below the converted minimum should not leave
the pending payment around as if a better invoice might still arrive through
that path. The test now checks the error, the removal, and the `PaymentFailed`
event together.

By the end, most of the reviewer gaps had concrete coverage and the remaining
ones were clearer because they did not fit the current shape as neatly.
Claim-side async rejection probably needs configurable test conversion state,
and end-to-end `InvoiceError` replies for static invoices still run into the
missing reply path. That is a better place to stop than pretending every gap is
the same size. The branch now has tests around the edges it can currently own,
and the follow-up work has a sharper outline.
