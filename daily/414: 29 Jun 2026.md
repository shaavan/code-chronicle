# Log 414: When a Channel Becomes a Path

> 29 June 2026

## What I worked on

- [Continued] the static-invoice refresh branch by strengthening the
  channel-opening test to prove that a replacement invoice gains the new
  payment path.
- Added coverage for forwarding-base-fee changes and helpers that find
  `ServeStaticInvoice` messages among interleaved offer-path requests while
  checking that each replacement stays in its original offer slot.
- Fixed a full-suite splice regression by collecting peers and usable channels
  only after confirming that async receiving has static-invoice-server paths to
  refresh.
- Traced an intermediate one-path replacement to `ChannelReady`, which fired
  before the new private channel had counterparty forwarding information.
- Removed that early trigger and its unused wrappers so the first applied
  forwarding update produces one replacement with the complete path set.

## Reflection

The focused tests had already shown that channel changes caused replacement
invoices. Today started with a more demanding question: did those replacements
contain the channel information that made the refresh necessary? Checking for
the extra payment path and its changed forwarding base fee made the tests
follow the data inside the invoice instead of stopping at the outgoing message.

The full suite then found two places where the implementation was acting too
early. The first was a splice reserve test with a channel in a transitional
state. Forced refresh gathered usable channels before checking whether the node
had any configured static-invoice-server paths, so a node that could not use
the result still entered `list_usable_channels()` and reached an unchecked
subtraction. Moving that collection behind the configuration check kept the
refresh path away from channel snapshots it had no reason to inspect.

The second failure was quieter. `ChannelReady` requested a refresh before the
new private channel had received its counterparty forwarding policy. That
produced a valid replacement containing only the old path, followed by the
useful two-path replacement once `ChannelUpdate` arrived. Channel usability was
not enough here. Until the forwarding fee and CLTV data exist, the channel
cannot become the blinded payment path the invoice needs to publish.

Removing the early trigger made the behavior smaller and more exact. A newly
opened channel now enters the invoice on its first forwarding update, and the
server receives one meaningful replacement in the same cached offer slot. The
focused tests explain what changed inside that invoice; the full suite fixed
when the implementation is allowed to look and when it has enough information
to send.
