# Log 399: The Wrong Edge

> 06 June 2026

## What I worked on

- Reviewed PR [#4659] against issue [#2348], starting from the issue so I
  could judge whether the patch was solving the right logging problem.
- Traced the relevant sync paths, including block sync, transaction sync,
  reorg handling, and filter registration, to find where LDK first has enough
  context to say that some chain activity is relevant to it.
- Reframed the logging around the sync layer instead of block connection
  callbacks, so confirmed transactions, reorg-side unconfirmations, and filter
  updates could be surfaced from one place without reporting every transaction
  in a connected block.
- Wrote the [review] around that point, calling out missing reorg-side
  unconfirmation logs, missing filter-registration visibility, and the need to
  summarize larger batches instead of logging every transaction one by one.

## Reflection

At first this looked like a simple review. The PR added logging, the issue
asked for more visibility, and it seemed like the main job would be checking
whether the messages were sensible. That feeling lasted only until I read the
issue before the diff.

The patch was looking at block connection, but the issue was really about
transactions LDK has a reason to care about. Those are close, but not the same.
A block callback can tell you what arrived in the block. It cannot as naturally
tell you which transactions became relevant, which ones became irrelevant again
during a reorg, or when filter state changed around them. Once I walked block
sync, transaction sync, reorg handling, and filter registration together, the
sync layer kept pulling the work back toward itself.

After that, I was no longer reading the patch as "more logs here." I was
reading it as "what edge of the system should speak first." Starting from the
wrong edge would still produce output, but it would keep needing exceptions,
deduplication, and after-the-fact explanation. By the time I left the review,
that felt like the real thing worth saying.
