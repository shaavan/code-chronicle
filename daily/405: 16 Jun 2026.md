# Log 405: Passing for Ordinary

> 16 June 2026

## What I worked on

- Continued the dummy-hop fee work on the `dummy-follow-22` branch by
  extending the receive-side test helpers so functional tests can optionally
  assert `dummy_skimmed_fees_msat` on both `PaymentClaimable` and
  `PaymentClaimed`.
- Kept those new assertions opt-in, which let the dummy-skim accounting become
  verifiable at the event surface without forcing unrelated receive-path tests
  to absorb the new field.
- Reworked blinded-path relay construction so real-hop relay fees are rounded
  into coarse buckets before landing in `PaymentRelay`, instead of carrying
  through exact values that would stand out too clearly.
- Updated router-built dummy relay TLVs to reuse a nearby low-fee profile when
  one exists, or fall back to common-looking network values instead of obvious
  zeroed defaults.
- Finished by keeping the privacy camouflage and the new accounting checks
  aligned, so the feature now reads as something that can be tested without
  making dummy hops look synthetic.

## Reflection

After the heavier receiver-side accounting work, this felt like a day where
dummy-hop fees had to stop existing only as internal correctness and start
holding up against two more practical questions: can I verify them cleanly, and
do the relay parameters still look believable once the feature is wired
through?

The test-helper changes were straightforward in shape, but they mattered
because they set a boundary for what "done" means here. It is not enough for
`dummy_skimmed_fees_msat` to appear somewhere deep in receive flow. It has to
show up in a way tests can ask for directly, without turning every existing
claim assertion into an argument about dummy-hop fees.

The relay-parameter side had more bite. Dummy payloads were still carrying
little tells: exact fee values when a real hop would look coarser, or fallback
defaults that practically announced themselves. Rounding real-hop fees into
buckets and letting dummy TLVs borrow nearby low-fee profiles made that code
feel closer to actual privacy work. By the end, the day was less about fee
accounting itself and more about whether this path could pass for ordinary
while still being easy to prove correct.
