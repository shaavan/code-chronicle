# Weekly Log 22: What Still Holds

> 29 May 2026 – 01 June 2026

## Overview

This stretch split across two different threads, but both of them kept asking
the same question: once the surrounding structure changes, what part of the
work is actually solid enough to keep.

On the currency side, that meant rebasing the branch onto the newer offers
tree and letting current interfaces decide which parts of the older patch
shape were still real. On the recurrence side, it meant backing out of
implementation momentum once the spec itself started looking too loose to rely
on. By the end, the useful progress was not mainly more code. It was getting
stricter about which assumptions deserved to survive contact with the current
tree or with the design underneath it.

## Highlights

### Forcing the currency branch through the current tree

- Rebased the currency series onto the newer offers stack and rethreaded the
  work through current interfaces instead of preserving the older patch layout
  by force. ([Branch][Currency])

- Updated the surrounding `ChannelManager` paths, helpers, fuzz targets, and
  tests to match newer offers changes such as the `BestBlock` to
  `BlockLocator` shift, so the branch sat on present code rather than on
  compatibility shims.

- Tightened the currency model itself by making `ExchangeRate` compare by
  value, putting `ExchangeRange` behind checked construction and accessors, and
  splitting converted-amount failures more honestly between unverifiable,
  below-minimum, and above-maximum cases.

- Adapted the series to metadata-carrying offer flows so invoice builders,
  refunds, invoice-request contexts, static invoices, and blinded payment
  paths all keep `payment_metadata` inside the main offers path.

### Turning recurrence back into a design problem

- Stopped treating the existing recurrence wording as fixed implementation
  ground and reworked the design at the spec level where the real ambiguity was
  coming from.

- Removed the security problem created by holding `payer_signing_pubkey`
  constant across recurring payments and folded that correction back into the
  recurrence model itself.

- Simplified the recurrence offer shape so it no longer leans on awkward
  parsing branches around optional versus compulsory fields or on whether a
  base time happens to be present.

- Reduced how much runtime interpretation the implementation would need just to
  decide what kind of recurrence object it is looking at, which made the
  remaining complexity feel more like recurrence logic and less like document
  cleanup.

## Reflections

What tied these days together was how quickly work improved once I stopped
protecting older shapes from present pressure.

The currency rebase made that visible first. It would have been easy to treat
the newer offers tree as something to patch around until the old series
compiled again, but that would only have preserved decisions that belonged to
the earlier base. Reworking the surrounding paths forced a harder standard.
If a type still trusted callers too casually, or if metadata only fit by
special handling, that was not rebase noise. That was the branch being asked
whether it actually understood the architecture it wanted to live in.

The recurrence work exposed the same weakness from the other side. There, the
code was no longer the main resistance. The spec was. Once the wording started
asking parser and implementation to recover intent from too many local cases,
continuing to code forward would only bury that weakness deeper. Pulling the
work back into the design made the rest of it feel more defensible
immediately. Complexity did not vanish, but it started belonging to recurrence
itself instead of to loose wording and prototype habit.

That leaves both threads in better shape for next pass. Currency work now has
to justify itself against current offers boundaries, not against memory of old
branch. Recurrence work now has to grow out of design that can carry its own
meaning before implementation starts adding weight on top of it.
