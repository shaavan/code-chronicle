# Log 373: Making the Checks Agree

> 30 April 2026

## What I worked on

- [Continued] the recurrence work on the `rust-recurrence` branch by auditing
  the top `InvoiceRequest` commits and fixing the drift left behind by earlier
  cherry-picks.
- Repaired stale `InvoiceRequest` recurrence validation and test fixtures so
  the stack was compiling cleanly again, then reran `cargo test -p lightning
  --lib --no-run` and `cargo +1.75.0 fmt --all`.
- Reviewed the current implementation against the recurrence draft and found
  several real mismatches: `Compulsory(None)` semantics, builder-side invalid
  recurrence combinations, dropped recurrence data, limit handling, and an
  under-specified `recurrence_paywindow` TODO.
- Consolidated recurrence semantic checks into
  `check_recurrence_semantics()` and wired them into `build_with_checks` so
  the builder now rejects the same invalid combinations as the parser.
- Corrected `RecurrenceLimit` handling so the value is treated as a count
  rather than a zero-based maximum, and updated the surrounding `offer.rs`
  docs to say that plainly.

## Reflection

Today started as stack maintenance, but it only became useful once I stopped
treating "compiles again" as the same thing as "holds together again."

The first fixes were mechanical: old `InvoiceRequest` validation was still
looking for a recurrence shape that no longer existed, and some fixtures had
been left behind with it. That kind of drift is easy to clean up and just as
easy to overvalue. The more important part of the day was the audit that came
after the build was green, because that is where the implementation stopped
looking merely patched and started showing where its semantics were still
uneven.

Most of the problems had the same feel. The parser knew things the builder was
still willing to let through, and some recurrence rules were being carried only
partway through the request path. `Compulsory(None)` had been blurred into the
optional case. The recurrence limit logic was off by one in a way that would
look small right up until someone tried to rely on it. Pulling those checks
together into one place made the code feel less split-brained. The builder and
parser now answer the same question with the same rules.

The unresolved part was useful too. `recurrence_paywindow` is still deferred,
but the TODO now says why: that check needs trusted current time and, in one
case, prior accepted recurrence state that this layer does not have yet.
Leaving that gap named precisely felt better than pretending it was one more
small validation fix waiting its turn. By the end of the day, the stack was not
just building. It was stricter about what it knew, and clearer about what it
still could not know.
