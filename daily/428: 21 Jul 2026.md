# Log 428: Real Constraints

> 21 July 2026

## What I worked on

- [Continued] the dummy-hop follow-up branch by making generated dummy-hop
  payment constraints match the relay realism added earlier.
- Changed dummy TLV construction so constraints are derived backward from the
  payee's downstream max CLTV and each dummy hop's CLTV delta.
- Placed the constraint fix in the commit that owns realistic dummy-hop relay
  parameters after checking the full PR range.
- Removed `DummyTlvs::default()` and the old default dummy constraints so a
  complete dummy TLV cannot exist without downstream context.
- Reworked the blinded-payment test helper to pass only the local dummy-hop
  count, while keeping `default_relay()` as a deterministic relay-policy
  fixture.

## Reflection

The dummy-hop branch had already made the relay parameters look more like real
route state, but review found one remaining place where the dummy hops were
still too easy to tell apart. The forwarding constraints were permissive in a
way no real hop would usually be: `max_cltv_expiry = u32::MAX` and
`htlc_minimum_msat = 0`. Since dummy hops go through the same blinded
forwarding validation as real hops, those defaults were not just ugly test
values. They were behavior.

The first version fixed the behavior by overwriting constraints after dummy
TLVs were generated. That worked, but it left the code feeling like it had
created the wrong object and then tried to make it acceptable afterward. The
better shape was to keep generated dummy hops as relay policies, then let
`dummy_tlvs_from_relays` build complete `DummyTlvs` only once it has the
downstream constraints needed to do so. Working backward from the payee's max
CLTV through each dummy hop's delta made the dummy path follow the same
accumulation pressure as real blinded forwarding.

That also made the commit boundary easier to defend. The underflow-label and
receive-side dummy-fee commits were carrying different concerns. The constraint
fix belonged with the commit that had already changed generated dummy-hop
relay parameters, because both pieces answer the same review question: whether
dummy hops behave like plausible hops when forwarding validation looks at
them.

The larger cleanup came from asking whether `DummyTlvs::default()` still made
sense. Once complete dummy TLVs need downstream constraints, a default
complete TLV is no longer a harmless fixture. Removing it pushed some awkward
test-helper code into view, especially around `PassAlongPathArgs`, but the
awkwardness pointed at the real invariant. The pass-along helper only needs to
know how many locally peeled dummy layers exist. It does not need to carry
dummy TLVs whose constraints were invented outside the route context.

The final test shape is less convenient, but more honest. Tests can still
build deterministic relay policies with `DummyTlvs::default_relay()`, then
derive actual dummy TLVs through the same helper production code uses. The
pass-along path receives a count instead of a fake object. That leaves the
branch with fewer shortcuts hiding in test support, and it makes the privacy
claim rest on the same data the forwarding code actually checks.
