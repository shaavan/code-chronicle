# Log 422: Positive Bounds

> 10 July 2026

## What I worked on

- [Continued] the currency-payment branch by making
  `payable_amount_msats()` the canonical conversion check for invoice response
  amounts.
- Moved `InvoiceBuilder::amount_msats()` onto the shared payable-amount path,
  preserving reference-rate quoting for requests without an explicit amount.
- Closed zero-bound conversion cases where absolute lower tolerance or
  exchange-rate flooring could make currency amounts accept or produce `0
  msats`.
- Removed an unnecessary converter lifetime from invoice response builders so
  callers can pass short-lived conversion snapshots.
- Added changelog coverage for the new public amount-error variants exposed by
  currency-denominated offers.

## Reflection

Yesterday's work made invoice rejection land on the right pending payment.
Today pulled the same currency branch back toward the arithmetic that decides
whether an amount can be signed in the first place.

The awkward part was that `InvoiceBuilder::amount_msats()` had started to
answer a question that already belonged to `payable_amount_msats()`. It was
not completely wrong, but it was narrower than the converted payable-amount
logic and easier to drift from the rules around lower bounds and maximum msat
values. That is a risky shape for currency offers because invoice responses
are one of the last places where a bad explicit amount can still become a
signed invoice.

Making `payable_amount_msats()` the shared path tightened that boundary. The
helper still keeps the old behavior for omitted request amounts, where the
payee uses the reference rate to quote the invoice. Explicit amounts now pass
through the same converted-minimum and `MAX_VALUE_MSAT` checks before the
builder signs anything. The code became less interested in which caller found
the amount and more precise about the one question that matters: can this
currency amount become a payable msat value?

The zero cases had a different feel. They were small numeric edges, but they
cut straight through the underpayment protection. An absolute lower tolerance
that erased the exchange rate left a minimum bound of zero, and a later
conversion rate could floor a non-zero currency amount into `0 msats` even
after offer construction had rejected zero amounts. Moving those checks into
the conversion boundary made the invariant local: a converted amount must stay
positive, and a lower accepted rate must stay positive, or the rest of the
validation is pretending to measure something.

The API cleanup was quieter but still useful. The converter passed into
invoice response builders is only needed while computing the amount, so tying
its borrow to the returned builder made callers hold a lifetime that the code
did not actually use. Loosening that signature keeps the builder easier to
compose with short-lived rate snapshots. The changelog update belongs in the
same cleanup: once new amount failures are public enum variants, downstream
callers need to see that surface change clearly.

By the end, this part of the branch felt less split across callers. Amount
validation, zero rejection, and the public API all point at the same narrow
line now: conversion may be policy-driven, but the msat value that comes out
has to be positive, bounded, and checked in one place before it is trusted.
