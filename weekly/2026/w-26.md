# Weekly Log 26: Where the Decision Holds

> 23 June 2026 – 03 July 2026

## Overview

This stretch moved across several branches, but each one became easier to reason
about after finding the point where a decision could be made once and then
carried forward.

The static-invoice work began with stale payment paths and grew into a more
precise account of what can make an invoice stale, when a replacement has
enough channel information to be useful, and who decides how many replacements
the server will store. The currency branch followed amount validation through a
similar correction. Conversion policy had drifted into invoice verification;
it now sits at the outbound payment boundary, before payment state exists. A
review of the dummy-hop stack then tested whether its small relay fees remained
coherent through capacity limits, persistence, events, and test expectations.

Most of the difficult work came from plausible boundaries that were slightly
too early or belonged to the wrong owner. Channel readiness arrived before
forwarding policy. Commitment feerates sounded relevant to invoice refreshes
without changing anything the invoice contains. Invoice authentication could
see an amount without owning the decision to pay it. Following the actual data
made each path smaller.

## Highlights

### Refreshing static invoices from usable channel state

- Added an event-driven refresh path for published and pending static invoices,
  with channel transitions only marking work while invoice construction stays
  outside lock-heavy processing and reuses the timer-refresh builder.

- Moved routing-related refreshes from commitment-feerate changes to applied
  `ChannelUpdate`s, where forwarding fees and CLTV deltas used by blinded paths
  actually change.

- Strengthened coverage beyond replacement-message emission to verify that new
  channels and forwarding-policy changes appear inside the replacement invoice
  while preserving its original offer slot.

- Removed the early `ChannelReady` trigger after full-suite failures showed it
  could rebuild before counterparty forwarding information existed, leaving the
  first applied forwarding update to produce one complete replacement.
  ([Refresh PR][static-refresh])

### Letting the server define active capacity

- Traced the fixed ten-slot allowance across recipient-side offer creation,
  static invoice persistence, and the server application's storage boundary.

- Put a configurable non-zero allowance in authenticated blinded-path context
  and returned it through `OfferPaths`, while keeping absent fields compatible
  with the existing ten-slot behavior.

- Enforced the allowance on incoming slot numbers and recipient cache activity,
  then persisted it with a legacy decoding rule derived from the old offer
  vector where possible.

- Kept offers above a reduced limit stored until their published details expire
  while excluding them from return and refresh, separating current capacity
  from promises already handed to users. ([Branch][server-config])

### Moving currency policy to payment construction

- Made recipient-side conversion fail closed when the converted amount cannot
  be verified, and expanded `ExchangeRateBound` coverage around fractional
  tolerances, invalid bounds, and overflow.

- Added async static-invoice coverage proving that a currency-denominated offer
  retains its converted payment amount separately from routing fees placed on
  the outbound HTLC.

- Returned explicit `invoice_error` responses for unverifiable, insufficient,
  and excessive invoice amounts instead of leaving those outcomes silent.

- Separated invoice authentication from payment policy by passing conversion
  into outbound payment handling, where rejection now occurs before payment
  state or failure events can be created. The bindings-friendly public surface
  for that converter remains open. ([Branch][currency])

### Reviewing dummy-hop costs end to end

- Finished the earlier dummy-hop CI and history cleanup, then reread the larger
  follow-up from retained-fee accounting through realistic relay parameters.

- Reworked fee buckets around common network values with bounded high-fee
  rounding, and kept dummy relay policies low and independently selected.

- Preserved downgrade compatibility by leaving the original HTLC map in its
  even TLV and adding dummy-skim metadata through a new odd TLV that newer
  decoders can reconnect by channel and HTLC ID.

- Followed conservative sender overpayment through async, retry, BOLT 12, and
  MPP expectations while keeping dummy-hop fees inside the capacity advertised
  to the payer. The high-fee disable threshold still needs reviewer agreement.

## Reflections

The static-invoice branch carried most of the movement in this stretch. It
started with a direct requirement: payment paths published in a static invoice
should react when channel state makes them stale. The first implementation
already avoided rebuilding under channel locks, but review and full-suite
coverage kept narrowing which state changes belonged in the trigger. A Bitcoin
commitment feerate never enters the invoice. `ChannelReady` says a channel can
be used, yet arrives before the counterparty forwarding values needed to build
its blinded path. Once those events were removed, the remaining refresh points
matched information the invoice can actually publish.

The storage allowance extended that work across the application boundary. Ten
was previously chosen where the recipient allocated its cache, even though the
server application bears the persistence cost. Sending the server's
authenticated allowance through the existing offer-path exchange gave both
sides the same number before invoices were created. Reducing that number then
exposed a second boundary: active capacity can change immediately, while an
offer already published still needs to resolve until expiry. Keeping those
entries dormant rather than deleting them lets the new configuration govern
future work without breaking old payment details.

Currency conversion needed the same kind of ownership correction. Amount
checks initially moved alongside invoice authentication because that code had
the necessary data. It still did not own the consequence. The outbound payment
path decides whether state will be created and which amount will be paid, so
conversion policy now runs there. Unsupported or excessive amounts stop before
they leave payment state behind, while authentication remains concerned with
whether the invoice belongs to the payer's request.

The dummy-hop review was a useful final check on this way of working. A fee of a
few millisatoshis touches route construction, recipient accounting, advertised
capacity, persistence, and event assertions. Each layer now carries only the
piece it owns: older readers retain the HTLC map they understand, newer readers
can restore skim metadata, and sender overpayment stays distinct from the
amount claimed by the recipient. Across all three threads, the code held once
the decision moved to the boundary that also owns its consequences.
