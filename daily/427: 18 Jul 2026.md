# Log 427: Ready Can Be Stale

> 18 July 2026

## What I worked on

- [Continued] the async static invoice refresh branch by tightening which
  cached invoices get rebuilt after channel state changes.
- Changed forced refresh selection to include every cached offer with a
  server-side static invoice, including entries still in the `Ready` state.
- Updated the refresh documentation so staleness is tied to channel state
  changing underneath stored invoices, not to offer age or application use.
- Replaced the local `flow.rs` macro for `ServeStaticInvoice` update
  construction with a shared closure.
- Fixed two async payment tests so they allow unrelated `OfferPathsRequest`
  setup messages while still rejecting any inline `ServeStaticInvoice` update
  before the timer tick.

## Reflection

After yesterday's work separated direct close refreshes from batched peer
updates, the next weak spot was smaller but still important. The forced refresh
path was trying to reason about cache state. It skipped `Ready` offers because
they looked unused, and unused offers seemed safe to leave for normal
rotation.

That assumption did not quite hold once I looked at the cache from the
application boundary instead of from inside the refresh code. A ready entry is
still one call away from being handed out. If it already carries a server-side
static invoice built from older channel state, it can be stale before anyone
marks it used. The cache is small enough that being selective there did not buy
much, and it made the rule harder to defend.

The better version was more direct: if a cached offer has a server-side static
invoice, forced refresh includes it. The offer's internal state no longer has
to answer a question it cannot really answer. Staleness comes from the channel
data the invoice was built from, so the selector should follow that boundary.
The documentation became easier to write once the code stopped depending on a
distinction that only looked useful from one angle.

The `flow.rs` cleanup had the same shape in miniature. The macro had been
local and harmless, but a closure expressed the shared construction without
making the call sites feel like they were inside a little language of their
own. Timer-driven refreshes and forced refreshes still build the same
`ServeStaticInvoice` update; the important difference is only which cached
offers reach that construction.

The tests were where the change showed its edges. Including ready offers meant
the setup path could leave unrelated async-offer cache refill messages in the
onion messenger queue. The old assertion expected that queue to be empty before
the explicit timer tick, which was broader than the behavior under test. The
real invariant is narrower: peer updates should not send a static invoice
refresh inline, and the timer tick should send the coalesced update later.

The first instinct was to drain setup messages, but that would have hidden too
much. The final helper peels the pending messages and only permits unrelated
`OfferPathsRequest`s. If a `ServeStaticInvoice` update appears early, the test
still fails. That leaves the tests less brittle without making them less
strict about the thing that matters.

By the end, the branch had fewer clever boundaries. Ready offers can still be
stale. Queue emptiness is not the same as no refresh. The refresh code now
tracks the channel-state dependency directly, and the tests check the batching
contract without depending on incidental setup silence.
