# Log 383: One Shape to Carry

> 14 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  replacing the older scattered `InvoiceRequest` recurrence fields with one
  explicit model for the two cases the protocol actually cares about:
  basetime-anchored requests and streams that begin from the first successful
  payment.
- Defined the recurrence forms more tightly so each one carries only the data
  that is real for that path: the basetime variant keeps both counter and start
  offset, while the non-basetime variant stops pretending there is a meaningful
  `start` to preserve.
- Updated the offer-to-request builder path, TLV conversions, and
  parsing/serialization helpers so recurrence now moves through the request
  stack as one structured piece of state instead of a loose cluster of optional
  metadata.
- Threaded the provisional request-side recurrence token through that same path
  as an echoed opaque value, keeping the plumbing in place without teaching the
  local implementation to interpret it too early.
- Aligned the surrounding wire-level paths, including payment-context
  extraction, refunds, and fuzz/test scaffolding, so shared TLV handling agrees
  on the recurrence shape even where a given message type should still reject
  it in practice.

## Reflection

Today mostly came down to making the request stop lying about what kind of
recurrence it was carrying.

The older shape had too many optional pieces that only made sense in some
cases, which meant the code could assemble combinations that looked flexible
but were really just vague. Once I split the model along the line the protocol
already cares about, the rest of the stack got easier to read. A request tied
to an explicit basetime has different obligations from one whose cadence only
starts after the first successful payment. Treating those as the same thing had
been forcing too much interpretation into places that should only be carrying
state.

What I liked here was that the improvement was not isolated to one struct. The
builder path, TLV layer, and request helpers all started carrying the same
shape, so recurrence stopped feeling like special knowledge hidden inside
invoice-request internals. By the end of the day I could point at one request
model and see the rest of the stack beginning to hold onto it without dropping
or inventing pieces along the way.
