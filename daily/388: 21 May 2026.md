# Log 388: No Fake Precision

> 21 May 2026

## What I worked on

- [Continued] the BOLT 12 currency work on the `currency-99` branch by making
  `CurrencyConversion` an application-supplied dependency instead of letting
  LDK invent its own exchange-rate authority.
- Changed the conversion contract to return an accepted millisatoshi range
  rather than one exact quote so fiat-denominated offers can carry uncertainty
  honestly instead of flattening it into fake precision.
- Threaded that conversion state through `ChannelManager`, offer and invoice
  construction, and the surrounding test and fuzz setup so currency amounts can
  move through normal offer flows instead of only existing at the type level.
- Opened `OfferBuilder` up to public currency-denominated amounts and kept the
  validation path strict enough to reject invalid converted ranges before they
  can sneak around the existing bitcoin amount limits.

## Reflection

Today was the first time the currency work started to feel honest.

The important decision was not really about one API surface. It was deciding
that LDK should stop pretending it knows what a fiat price is. Exchange rates
belong to the application, and even then they usually belong as a bounded
quote, not as one magically precise answer. Returning an accepted range made
the feature feel much less polished in the cosmetic sense, but much more
truthful in the protocol sense.

Once that held, the rest of the work became straightforward in a useful way. A
currency amount could not live as a clever type and still count as finished.
`ChannelManager`, builder paths, invoice paths, and the surrounding scaffolding
all had to carry the same conversion state or the feature would still collapse
back into millisatoshi-only assumptions the moment it touched real flow code.

What I liked by the end was that the change narrowed the promise instead of
widening it. The system does less pretending now. It can say that an offer is
priced in currency, that conversion comes from outside, and that the result is
only valid inside a bounded range. That felt sturdier than forcing the code to
sound more certain than the inputs actually are.
