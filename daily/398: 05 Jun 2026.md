# Log 398: State Between Peels

> 05 June 2026

## What I worked on

- [Continued] the dummy-hop fee-accounting work on the `dummy-follow-23`
  branch by tracing how skimmed fees could survive across multi-hop dummy-path
  processing on the receive side.
- Isolated the real blocker in the current design: dummy hops reuse the same
  queued forwarding flow as ordinary hops, which keeps processing uniform but
  discards local context between successive onion peels.
- Explored recursive peeling as one possible way to preserve fee state, then
  ruled it out because different dummy-hop counts would start to create
  different processing and timing shapes.
- Developed a cleaner direction based on receiver-generated, HMAC-authenticated
  continuation state that can carry dummy-hop accounting forward across queued
  receive-side steps.

## Reflection

Today moved the dummy-hop fee problem out of vague discomfort and into one
specific architectural boundary.

At first it looked like I was dealing with accounting logic: how to add up the
fees earned by dummy hops before the payment is finally claimed. But after
tracing the receive path more carefully, the harder problem was that the
current flow has no stable place to keep that information once processing stops
being one continuous local action. Dummy hops intentionally pass through the
same queued forwarding machinery as ordinary hops, which is good for keeping
the path behavior uniform, but it also means each peel forgets what happened in
the previous one.

I spent some time testing whether recursive peeling could avoid that break in
state, and it helped mostly by showing why that direction should not survive.
It would keep the context alive, but at the cost of making paths with
different numbers of dummy hops behave differently. That felt like fixing the
bookkeeping by weakening the privacy story.

What held up better was to leave the queued model alone and make the missing
context explicit. Once the problem was stated that way, the continuation-state
approach started to feel natural: carry the receive-side dummy-hop accounting
forward in authenticated state, update it at each peel, then verify and use it
when the real receive hop is finally reached. By the end of the day, the work
felt less like I was circling a tricky edge case and more like I had found the
exact seam in the design that needed a new piece.
