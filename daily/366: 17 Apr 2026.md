# Log 366: Where the Decision Waits

> 17 April 2026

## What I worked on

- [Continued] the recurrence transition work on the `rust-recurrence` branch.
- Pulled the inline inbound `InvoiceRequest` path apart inside `ChannelManager`
  so verified requests can now be handled manually through a config gate,
  persistent event, and public accept/reject APIs.
- Added async coverage for both manual response and rejection flows so the new
  delayed-decision path is exercised as behavior rather than left as exposed
  surface.
- Opened the payer side by adding public invoice-request builder and send
  primitives, including explicit payer-signing-pubkey and unsigned-builder
  paths for external signing.
- Cleaned up the branch stack after formatting drift and a bad reorder, then
  recovered the dropped commits, resolved the conflicts, and revalidated the
  final tree.

## Reflection

Today was the first time the recurrence move felt complete from both sides of
the request flow.

Up to this point, the architectural argument was strong, but too much of it
was still only an argument. The payee-side work changed that. Once
`ChannelManager` could surface a verified `InvoiceRequest`, persist it as an
event, and wait for a higher layer to decide whether to answer or reject it,
the boundary stopped being theoretical. There was now an actual pause in the
protocol flow where ownership could change hands without losing correctness.

The payer-side work mattered for the same reason. A higher layer can now build,
sign, and send an invoice request itself, including the externally signed path,
without giving up the payment tracking and invoice-authentication machinery
that still belongs inside `rust-lightning`. Refactoring `pay_for_offer*` onto
that lower-level send path made the abstraction feel more honest. The public
surface was no longer a side door around the real implementation. It had become
the implementation.

The tests were important because they forced those boundaries to prove
themselves end to end. One wrong turn in the explicit-signing case made that
obvious: the first version was trying to validate a recipient-side property
that was beside the point. Fixing that tightened the contract back to what
actually mattered, which was that an externally prepared request could move
through the flow and still be answered correctly.

The branch cleanup at the end was a different kind of work, but it belonged to
the same day. Reordering the stack, recovering the temporarily lost commits,
and putting the tree back into a sane state was what turned this from "the APIs
exist" into "Phase 1 on the `rust-lightning` side is actually done." By the
end, the branch had the shape I had been trying to reach all week: protocol
surface in `rust-lightning`, recurrence control ready to move upward.
