# Log 384: No Room for Hybrids

> 15 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  tightening `InvoiceRequest` recurrence validation so the request form now has
  to match the offer contract directly instead of passing through loosely typed
  combinations.
- Made the basetime split enforceable in both parsing and builder validation:
  offers with an explicit recurrence basetime now require the basetime-anchored
  request form, while offers without one must stay on the non-basetime path.
- Closed a follow-up state leak in initial recurring requests by rejecting
  cancellation and recurrence-token echoing when `counter == 0`, which keeps
  first requests from smuggling in fields that only make sense later in the
  stream.
- Replaced the older scalar-style recurrence handling in
  `InvoiceRequestFields` with one recurrence object that now flows into payment
  context and downstream comparisons without being split back into separate
  pieces.
- Updated the surrounding offer, invoice, refund, and fuzz tests, then cleaned
  up the docs and comments so the basetime and non-basetime split is visible
  from the request API instead of being buried in validation details.

## Reflection

Today was mostly about making recurrence stop looking valid when it was really
just underspecified.

Yesterday got the request into one honest shape. That made today's loose edges
much easier to see. The parser and builder were still accepting mixtures that
technically fit the field layout but did not say one clear thing about how the
request related to the offer. Tightening those paths made the contract more
useful because the invalid cases now fail for the same recurrence reasons the
valid ones succeed.

The `counter == 0` cleanup mattered for the same reason. Initial recurring
requests had still been able to carry a little too much follow-up-only state
just because the TLVs existed. Cutting that off made the request lifecycle read
more truthfully. A first request should start the stream, not pretend it is
already in the middle of one.

What I liked here was how much quieter the rest of the stack became once
recurrence stopped arriving as separate scalar fragments. Carrying one request
recurrence object into payment context and the comparison paths made the code
feel less like it was repeatedly reconstructing the same idea. By the end, the
strictness did not feel ornamental. It felt like the request path had finally
stopped drifting away from the offer contract it was supposed to honor.
