# Weekly Log 19: Where the Proof Lives

> 06 May 2026 – 11 May 2026

## Overview

This week kept pulling the recurrence work away from local convenience and back
toward things the stack could actually prove.

I started by reviewing what the branch still did not own cleanly. That exposed
one problem in the public request-building surface and another in the
continuation-token work. The builder was trying to carry an explicit payer
pubkey path without also owning the signing step that makes that path safe. The
token path was still too close to a transport trick for recovering basetime.
By the end of the week, both had been tightened around a more useful rule:
recurrence state should only live in the layer that already has the context to
justify it, and the token should authenticate recurrence membership rather than
just smuggle one value forward.

## Highlights

### Making signer ownership visible

- Reworked the cherry-picked explicit payer signing pubkey support so the
  privacy-first derived-key path stays on the default builder, while the
  explicit pubkey path now ends at an unsigned `InvoiceRequest` that the caller
  must sign themselves. ([Branch][Recurrence])

- Cleaned up the surrounding API, public formatting, and commit surface so the
  feature now describes the real boundary it opens instead of implying a
  broader safe-to-sign path than the code can honestly own.

- Added focused coverage for manually signing invoice requests built with an
  explicitly supplied payer pubkey, alongside compile checks and a full local
  `lightning` test run.

### Turning the token into authenticated recurrence state

- Replaced the earlier plaintext basetime-token idea with an authenticated
  token shaped as `encrypted_basetime || nonce || hmac`, bound to the offer,
  payer identity, and recurrence context the payee needs to trust later.

- Kept the crypto helper narrow by adding generic offer-bound encryption and
  verification support under `offers::signer` and `ExpandedKey`, while leaving
  recurrence-specific meaning in the `InvoiceRequest` layer instead of pushing
  it into flow logic too early.

- Used that pass to identify what still needed to become stricter: empty
  request recurrence tokens were still slipping through, and follow-up
  recurrence on basetime-less offers still depended on reaching back to
  `offer_recurrence_base`.

### Moving basetime and verification to the layer that has context

- Carried `expected_recurrence_basetime` through payer-side offer-payment
  context, retry-preserved state, and blinded reply-path handling so recurring
  invoice verification no longer has to invent knowledge during invoice
  parsing.

- Tightened the recurrence invariants around that state: follow-up recurring
  invoices must match the expected basetime exactly, while the first recurring
  invoice for a basetime-less offer is the only path that can proceed without a
  prior expectation.

- Moved the continuation-token ownership line into `flow.rs`, where token
  verification, `token_basetime` recovery, and effective basetime resolution
  can happen from full request context before invoice construction begins.

## Reflections

The week got better each time I stopped asking one layer to compensate for
missing context owned somewhere else.

That showed up first in the explicit payer pubkey work. The feature only looked
finished while I blurred over who actually owns the signing key. Once I held
that boundary still, the right shape was not hard to see. A caller can prepare
an invoice request with an explicit payer pubkey, but if that caller wants to
override identity directly, they also need to carry the final signing step
themselves.

The token work pressed on the same weakness from the recurrence side. A
basetime is part of the story, but it is not the full thing the payee needs to
trust. The token became much more solid once it started reading like
authenticated recurrence state instead of a place to stash one timestamp. That
change also made the remaining ownership problems easier to spot. Invoice
parsing could not honestly recover missing recurrence state on its own, and the
invoice layer should not have been the place deciding whether a follow-up
request actually belongs to the same stream.

By the end of the week, the stack felt less eager to look complete and more
willing to say what each layer really knows. The payer-side context now carries
the basetime expectations it is already responsible for. The request-flow layer
has a clearer place to verify recurrence tokens before invoice construction
starts. That left the branch narrower again, but in a way that feels usable:
less recurrence plumbing spread across convenient surfaces, more recurrence
state living where the surrounding proof already exists.
