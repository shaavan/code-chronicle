# Log 425: Common Enough

> 16 July 2026

## What I worked on

- [Continued] the dummy-hop follow-up branch by tightening the blinded-payment
  fee buckets after review feedback.
- Moved `dummy_skimmed_fee_msat` from even TLV `12` to odd TLV `11` so older
  versions can ignore the new event field on downgrade.
- Reworked deferred onion decoding persistence from a nested map into a flat
  list of skimmed amounts matched by position to TLV 14's pending HTLCs.
- Restored public `DummyTlvs` fields once it was clear the constructor did not
  enforce a stronger invariant than direct field construction.
- Resolved the `PaymentClaimable` rebase conflict and fixed a payer-proof offer
  test whose expected fee still missed the 20 msat conservative aggregation
  gap.

## Reflection

The dummy-hop follow-up came back through review today, and most of the work
was about making each piece no larger than the invariant it was carrying.
Nothing in the branch needed a new privacy idea. The pressure was in smaller
places: which fee values deserve a bucket, which TLV type can survive
downgrade, and how much structure persisted onion state actually needs.

The fee buckets were the first example. Calling something a common bucket only
helps if it stops preserving rare exact values. Keeping too many narrow cases
around `1`, `125`, `499`, and `501` made the tests precise in a way the design
was no longer trying to be. Reducing base fees to values like `0`, `100`,
`500`, `1_000`, `2_000`, and `10_000`, with the proportional buckets tightened
the same way, made the bucket table line up with its job again.

The TLV change was smaller, but sharper. `dummy_skimmed_fee_msat` had landed on
an even type, which means older versions would reject it instead of skipping
it. That is easy to miss when the new code reads fine, but downgrade behavior
is part of the wire shape too. Moving the field from TLV `12` to odd TLV `11`
kept the received-payment event extension compatible with older readers.

Deferred onion decoding took more care because the first storage shape was too
heavy. A map keyed by SCID and HTLC ID could find the value, but the data did
not need that much naming. The real invariant is positional: TLV 25's skimmed
fees must line up with TLV 14's pending HTLCs. That meant decoding TLV 14 as a
`Vec<(Key, Value)>` even though the wire bytes match a `HashMap`, because order
is the thing the later field depends on. After one reverted attempt, the final
version had count validation, better names, and comments only where the
positional rule would otherwise be easy to disturb.

The smaller cleanup around `DummyTlvs` had the same feel. Private fields only
help if the constructor protects something. Here it mostly nudged callers
toward `new`, without adding a real invariant, so making the fields public
again was less clever and more honest.

The rebase and test fixes were the end of the same thread. `PaymentClaimable`
needed to keep upstream's local `amount_msat` and
`counterparty_skimmed_fee_msat` while adding `dummy_skimmed_fees_msat`. The
payer-proof offer test then exposed one remaining place that still expected
the old 30 msat fee instead of the 50 msat total produced by conservative
dummy-hop aggregation. Updating it with the same 20 msat extra-fee expectation
as the nearby offer tests kept the accounting consistent.

By the end, the branch felt less like it had been patched and more like it had
been pared back to the shapes the data actually needs. Buckets are broad where
they are supposed to be broad. The new event field is safe for older nodes to
ignore. Persisted skimmed fees follow the pending HTLC order instead of
inventing another lookup table. Formatting and `git diff --check` passed; the
target test run was stopped before completion when asked not to run tests.
