# Log 369: Check without Checker

> 23 April 2026

## What I worked on

- [Continued] the recurrence design work by answering a question around
  verifying inbound recurrence invoices without depending on LDK Node during
  the invoice handling path.
- Worked out a way for LDK Node to provide recurrence session verification
  material when creating the invoice request, so LDK can carry that context
  through the blinded reply paths.
- Re-evaluated the earlier payer recurrence PoC and decided the production
  implementation should start from a cleaner LDK and LDK Node boundary instead
  of growing directly from that branch.
- Sketched the split between LDK-owned wire fields, APIs, and protocol
  handling, and LDK Node-owned session state, update logic, expiry handling,
  and recurrence policy.
- Turned the design around one constraint: the recurrence flow should not need
  mid-flow back-and-forth between LDK and LDK Node to know whether an invoice is
  valid.

## Reflection

Today brought recurrence back from a broad direction into one small protocol
question I could actually hold in my head.

The question was basically this: when the invoice comes back, should LDK already
have enough information to verify it, or should it have to ask LDK Node for the
recurrence session state at that point? The second answer felt awkward as soon
as I wrote it down. If every recurrence invoice needs a mid-flow check with LDK
Node, then the split between the two layers is still a little leaky in the one
place where I most want it to feel boring.

The nicer answer was to move the needed context earlier. LDK Node already knows
which recurrence session it is trying to advance when it creates the invoice
request. If it gives LDK that verification material up front, LDK can tuck it
into the blinded reply paths. Then, when the invoice comes back, LDK is not
standing there empty-handed. It can verify the invoice locally, while LDK Node
still owns the longer-lived session state and policy.

That made the old PoC feel useful, but also a little less precious. I do not
think I want to drag its exact shape forward just because it exists. It helped
me learn the terrain, but the production version should probably start from the
boundary I understand now: LDK carries the wire fields, APIs, and protocol
checks, while LDK Node carries session updates, expiry, and the meaning of a
valid recurring payment over time.

I liked this kind of planning day more than I expected. No code landed, but the
implementation stopped feeling like a pile of pieces from the previous branch.
It has a simpler rule now: give LDK enough to check what returns, and do the
rest of the recurrence thinking one layer up.
