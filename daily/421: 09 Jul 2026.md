# Log 421: Where Rejection Lands

> 09 July 2026

## What I worked on

- [Continued] the currency-payment branch by fixing terminal invoice amount
  rejections so they abandon the pending payment they belong to.
- Moved currency invoice validation after pending-payment lookup, preserving
  enough state to fail `UnverifiableAmount` and `ExcessiveAmount` cases
  directly instead of leaving them alive.
- Kept insufficient-amount handling separate from terminal amount failures
  where corrected invoice data or existing flow behavior can still resolve it.
- Documented the async static-invoice claim path as fail-closed when currency
  conversion is unavailable at claim time.
- Updated `InvoiceRequest::amount_msats()` Rustdoc and changelog text to point
  callers toward `payable_amount_msats()` for inferred payable amounts.

## Reflection

Yesterday's work tightened the places where converted amounts enter the
payment flow. The next loose edge was where the payment goes after validation
rejects an invoice.

The first bug was easy to miss because the caller already received an error.
`UnverifiableAmount` and `ExcessiveAmount` returned before the invoice had been
matched to its pending payment, so the visible rejection did not actually close
the state behind it. In the normal path, that payment could later surface as a
stale-timeout failure. In the manual-invoice path, it could sit in
`InvoiceReceived` without producing the `PaymentFailed` event the payer needed.

Moving the lookup before terminal validation made the failure less detached.
Once the invoice is associated with the pending payment, rejection has enough
context to abandon the right entry. That also narrowed the oracle concern:
without that ordering, a recipient could keep submitting invoices and observe
amount-validation responses without consuming the payment. After the change, a
terminal validation failure spends the pending opportunity it is testing.

The awkward part was keeping that behavior narrow. Insufficient amounts are
still different from unverifiable or excessive ones. They can be corrected or
flow through existing handling, so treating them as the same kind of final
rejection would make the state machine stricter in a place that does not need
it. The useful boundary was the payment's future: if the validation failure
means this pending payment cannot honestly continue, the rejection should close
it.

The async static-invoice case carried a different pressure. For requests
without an explicit amount, the payer commits to an msat value at send time,
but the recipient derives the expected amount again at claim time. Persisting
that expected value into the async offer context would avoid a live conversion
dependency, but it would pull persistence, blinded payment context encoding,
and compatibility into this pass. The branch now says the claim path fails
closed when conversion is unavailable, which is harsher for liveness but keeps
the underpayment boundary intact.

The public API cleanup carried the same pressure in documentation. Splitting
`amount_msats()` from `payable_amount_msats(&converter)` is the right model
for currency offers, but the old method name could hide a migration break for
callers that expected inferred amounts. The Rustdoc and changelog now make that
split explicit. By the end, the code and docs were both a little less willing
to leave meaning implied: rejection closes the right payment, claim-time
conversion fails safely, and callers have to ask for the payable amount when
that is what they mean.
