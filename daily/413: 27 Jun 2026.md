# Log 413: What the Invoice Knows

> 27 June 2026

## What I worked on

- [Continued] the static-invoice refresh work by investigating a report that
  fee-changing timer ticks could emit duplicate `ServeStaticInvoice` messages
  for overlapping pending and used offers.
- Built a single-pass timer fix around the existing normal and forced selectors,
  then traced the assumption that a commitment-fee change should force a static
  invoice refresh at all.
- Confirmed that `update_channel_fee` changes the Bitcoin commitment-transaction
  feerate, while static invoices depend on Lightning forwarding fees and CLTV
  deltas carried by channel updates.
- Removed the commitment-feerate trigger and moved routing-related refreshes to
  applied `ChannelUpdate`s that actually change a local channel's counterparty
  forwarding information.
- Reworked the feature commit and surrounding comments while preserving the
  valid channel-close, splice, deferred-refresh, and lock-safe processing paths.

## Reflection

Today started with what looked like a duplication bug. The normal timer refresh
and the forced refresh selected overlapping offers, so running both after a fee
change rebuilt and sent some invoices twice. The first fix followed directly
from that shape: choose one selector for the tick, since the forced selection
already contained everything the normal path would return.

That fix worked mechanically. It became harder to defend once I started asking
how often the trigger could fire and what those fee changes meant for the
invoice. `update_channel_fee` changes the feerate used for Bitcoin commitment
transactions. The blinded payment paths inside a static invoice care about a
different set of values: forwarding base fees, proportional fees, and CLTV
deltas. The shared word "fee" had made two separate parts of the channel look
more related than they were.

Once I followed the data in the invoice, the timer problem mostly disappeared.
A commitment-feerate update cannot stale a blinded payment path, so making its
forced refresh less repetitive would still leave the wrong event in charge.
The routing trigger now sits where an applied `ChannelUpdate` changes the
counterparty forwarding information used to build that path. Channel closes and
splices still invalidate real path state, and the deferred flag still keeps the
rebuild outside lock-heavy channel processing.

I liked that the review report led past its own symptom. The duplicate sends
were real. Following them far enough showed that commitment-fee updates should
not refresh static invoices in the first place. The branch now has fewer
refresh events, and each remaining one points to information the published
invoice can actually contain.
