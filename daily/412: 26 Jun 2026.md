## Reflection

Today was a quieter kind of work than yesterday. The branch already had code,
tests, and a cleaned-up stack, so the useful question was whether the whole
thing still made sense once I was no longer inside the implementation loop.
Static invoice refreshes touch enough pieces that a working patch is not quite
enough. The review path needs to show why each piece exists and where the
boundaries are.

The part I kept coming back to was the line between when an invoice should be
refreshed and how the replacement invoice is built. Channel opens, closes, and
fee updates can make published async receive invoices stale immediately, but
that does not mean channel-state code should start rebuilding invoices itself.
The PR now makes that argument directly: channel paths only mark that a forced
refresh is needed, and the actual `ServeStaticInvoice` construction stays on
the same path as the timer-based refresh.

That boundary also made the commit order matter. The cache selector is its own
step, then the flow plumbing, then the event trigger, then the test that shows
a new channel causing a replacement invoice for the same used-offer slot. The
shape is small enough that reviewers can inspect each pressure point without
having to infer the design from one large diff.

Opening the PR moved the work out of my local headspace. Yesterday was about
getting the lock ordering and refresh behavior to hold in code. Today was
about making the branch explain itself: static invoices depend on current
payment paths, timer freshness is too slow for channel changes, and the fix
should update published server-side invoices without creating a second invoice
builder or doing heavy work while channel locks are still involved.
