# Log 395: Spec Pressure

> 01 June 2026

## What I worked on

- Reworked the recurrence design at the spec level instead of continuing to
  treat the existing wording as fixed implementation ground, so the prototype
  stopped accumulating behavior around loose interpretation.
- Removed the security exposure created by keeping
  `payer_signing_pubkey` constant across recurring payments and folded that
  change back into the recurrence model itself.
- Simplified the recurrence offer shape so it no longer depends on awkward
  parsing branches around optional versus compulsory fields or on whether a
  base time happens to be present.
- Reduced the amount of runtime interpretation the implementation would need
  just to decide what sort of recurrence object it is looking at.

## Reflection

This day stopped feeling like prototype work pretty quickly.

I had gone in expecting to keep pushing the recurrence implementation forward,
but the more I touched it, the more the resistance came from the spec rather
than the code. Too much still depended on reading intent out of ambiguous
encoding rules. That made every implementation step feel weaker than it should
have, because even the correct path still looked partly like a guess.

So the work turned around. Instead of trying to code past that uncertainty, I
spent the day tightening the recurrence design itself. Removing the constant
`payer_signing_pubkey` was part of that, but the bigger gain was in stripping
down how many special cases the recurrence offer shape asked both parser and
implementation to carry around. Optional versus compulsory fields, base time
versus no base time, one recurrence form versus another: too much of the
complexity was coming from figuring out what the object even meant before the
actual recurrence logic could begin.

Once that started collapsing into a cleaner shape, the work looked more
serious again. The remaining complexity felt like recurrence complexity, not
spec-noise complexity. That mattered for implementation, but also for the
thesis. It is one thing to show that a prototype can be made to run. It is
another to present the design as something coherent enough that the code is not
quietly compensating for the document underneath it.
