# Weekly Log 16: Where It Can Wait

> 16 April 2026 – 17 April 2026

## Overview

This week was short, but it changed the recurrence reset from an architectural
position into a working surface.

I spent these two days rebuilding only the parts of the old PoC that still
belong in `rust-lightning`: recurrence-capable protocol objects, builder seams
for timing and signing, and the request-flow boundary where a higher layer can
decide whether an incoming invoice request should be answered at all. That made
the split with `ldk-node` feel much less speculative. The branch was no longer
defined by what had been removed. It was starting to show, in concrete API and
event shape, what remains once recurrence-specific lifecycle control moves up a
layer.

## Highlights

### Rebuilding the protocol surface

- Reintroduced recurrence-capable support across `Offer`, `InvoiceRequest`, and
  `Invoice` on top of the reset `rust-recurrence` branch. ([Branch][Recurrence])

- Added semantic validation coverage around that restored surface so the new
  objects were exercised as behavior rather than only reinstated as types.

- Refactored invoice creation around explicit `created_at`, keeping basetime as
  something higher layers can own instead of baking recurrence assumptions back
  into `rust-lightning`.

### Opening explicit handoff points

- Added explicit payer-signing-pubkey support and public invoice-request
  builder/send paths so higher layers can prepare and sign requests without
  losing the payment-tracking and invoice-authentication machinery that still
  belongs below.

- Pulled apart the inline inbound `InvoiceRequest` path in `ChannelManager` so
  verified requests can now pause behind a config gate, persist as an event, and
  be accepted or rejected through public APIs.

- Added async coverage for delayed response and rejection flows, including the
  externally signed path, so the new ownership boundary is exercised end to end.

### Finishing the reset cleanly

- Refactored `pay_for_offer*` onto the lower-level invoice-request send path so
  the public surface became the real implementation rather than a side route.

- Cleaned up formatting drift and a bad branch reorder, recovered dropped
  commits, resolved the conflicts, and revalidated the final tree so the first
  `rust-lightning` phase ended in a stable shape.

## Reflections

The reset branch finally stopped relying on explanation this week.

The earlier argument was that recurrence-specific lifecycle control does not
belong inside `ChannelManager`, but that claim only goes so far until the code
shows where responsibility can change hands without becoming vague or unsafe.
Rebuilding the protocol objects on the fresh branch helped narrow what
`rust-lightning` still needs to own. The more important step was making the
request flow actually wait. Once a verified `InvoiceRequest` could surface,
persist, and sit for an explicit accept-or-reject decision, the architectural
split had a real place to stand.

The payer-side work pushed in the same direction. Exposing builder and send
paths for externally prepared requests made it possible to move signing and
timing outward without turning the lower layer into dead scaffolding around a
special case. Refactoring `pay_for_offer*` onto that same surface made the
boundary feel honest.

The branch cleanup at the end mattered for more than neatness. This kind of
work becomes hard to trust if the stack itself feels unstable. Recovering the
lost commits, reordering the branch properly, and finishing on a validated tree
made the result feel settled enough to build on. By the end of the week, Phase
1 no longer looked like a reset in progress. It looked ready for the recurrence
control layer to move upward.
