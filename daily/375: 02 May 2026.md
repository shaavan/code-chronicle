# Log 375: The Missing Half

> 02 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  adding the request-side continuation-token plumbing needed for follow-up
  recurring `InvoiceRequest`s.
- Introduced `invreq_recurrence_token` as a clearly provisional odd TLV using
  temporary type `95`, and threaded it through `InvoiceRequest` storage,
  builder/accessor APIs, parsing, serialization, and recurrence validation.
- Kept the change scoped to invoice requests by updating the shared
  invoice-request/refund TLV plumbing just enough for the new field to compile
  cleanly without widening invoice behavior ahead of the rest of the proposal.
- Documented throughout the code that the token support is still speculative,
  not part of the upstream recurrence spec, and still missing the invoice-side
  completion work that would make the round trip real.
- Verified the feature end to end with the full `lightning` crate test suite,
  then added focused coverage for recurrence-token encoding and parsing on
  follow-up requests while rejecting token use on an initial recurring request.

## Reflection

The recurrence work has been getting stricter for a few days now, but the
useful question today was different. I was no longer trying to make the
existing request path more correct. I was trying to see whether I could add one
speculative piece without letting it pretend to be more settled than it is.

That made the boundaries matter more than the field itself. The token needed to
exist far enough through `InvoiceRequest` that the rest of the continuation
flow could be exercised, but stopping short in the wrong place would have left
the code with a fake sense of completeness. Making it an odd TLV with a
temporary type, and repeating in the surrounding docs that it is provisional,
helped keep that tension visible. The implementation now carries the idea
cleanly without trying to smuggle it in as accepted wire behavior.

The other part I liked was keeping invoice behavior out of it. There was an
easy version of this change that would have started spreading token assumptions
across the broader recurrence stack just to make the request side feel more
finished. Holding the work at the request boundary kept the change honest. By
the end of the day, I had a feature commit that was intentionally incomplete in
one precise way, plus tests that made the intended usage legible: follow-up
recurring requests can carry the token, but an initial request cannot invent
one out of nowhere.
