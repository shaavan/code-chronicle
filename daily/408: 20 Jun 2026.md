# Log 408: What the Limit Promises

> 20 June 2026

## What I worked on

- [Continued] the dummy-hop fee work on the `dummy-follow-29` branch by pulling
  amount accounting back out of the shared test framework and into the payment
  processing path that owns it.
- Preserved dummy skim across both immediate onion peeling and deferred
  `update_add_htlc` decoding, so the two paths now arrive at the same
  receive-side accounting without a second model in the harness.
- Reduced the remaining fee failures to narrow assertions for bounded upward
  rounding instead of teaching shared helpers to reconstruct dummy-hop amount
  flow.
- Traced the final async MPP regression to a real limit mismatch: the blinded
  path advertised the payee's full inbound HTLC limit before accounting for
  fees the sender would add for dummy hops.
- Corrected the advertised maximum by applying `amt_to_forward_msat` for each
  dummy hop, then verified the remaining MPP regression and blinded-payinfo
  tests.

## Reflection

Yesterday ended with one coherent class of failures. I expected today to be a
matter of finishing the helper work around them. Once I followed the amounts
through, that approach started to feel backwards. The test framework was
becoming responsible for understanding too much of the feature, while the
production path already had the right places to preserve the skim across both
immediate and deferred processing.

Pulling that responsibility back made the failures easier to read. A few were
small, deterministic effects of rounding fees upward so a path is never
underfunded. Those belong at the assertions where the difference appears. The
last MPP failure was different. The path was promising that the payee could
receive its full inbound HTLC limit, even though routing still had to place
dummy-hop fees above that amount.

That bug resisted changes to the test payment and channel limit because both
only moved the same boundary. The advertised amount itself had to leave room
for the fees that would be added later. Running the channel limit backwards
through each dummy hop finally made the promise match what the channel could
accept.

I liked where this left the branch. The tests no longer need a parallel account
of the payment, and the final red case turned into a useful check on the actual
contract exposed by the blinded path.
