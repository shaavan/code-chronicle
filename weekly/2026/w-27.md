# Weekly Log 27: Before the Amount Moves

> 07 July 2026 – 13 July 2026

## Overview

This week moved from the last dummy-hop accounting failures into the
currency-payment branch, where the work kept returning to the places an amount
becomes real enough to send, sign, claim, or reject.

The dummy-hop work started with noisy full-suite failures caused by randomized
relay policies in tests. Making test builds deterministic exposed the smaller
gap underneath: advertised aggregate policy is deliberately conservative, but
forwarding still applies each dummy hop's own rounded fee. Once the tests
described those as two different calculations, the remaining expectations
became predictable instead of a trail of adjusted constants.

Most of the week then stayed with currency conversion. The branch closed
underpayment paths before invoice requests are signed, before async claims are
accepted, and before invoice responses are built. It also made terminal amount
failures abandon the pending payments they belonged to, kept zero-valued
conversion edges from slipping through tolerance math, and filled reviewer
coverage around unsupported currencies, exact bounds, fuzzed conversion
inputs, and static-invoice failures. By the end, the branch needed more care
around review shape: notes grouped by surface area, and rebase repairs placed
in the commits that introduced the changed API contracts.

## Highlights

### Separating advertised and applied dummy-hop fees

- Made dummy-hop relay policies deterministic in test builds while preserving
  independently sampled policies for production path construction.

- Traced offer and MPP failures to the difference between conservative
  aggregate policy used for routing and per-hop rounded fees applied while
  forwarding.

- Updated recipient amount, sender fee, CLTV, retry, unannounced-introduction,
  small-payment, and MPP expectations around the two fee views.

- Reduced the reviewer update to the parts that shaped the revision:
  bucketing, randomized policy selection, downgrade-compatible serialization,
  and documentation. ([Branch][dummy-follow])

### Checking currency amounts before they travel

- Closed payer- and recipient-side underpayment paths for
  currency-denominated invoice requests before a signed request or async
  keysend claim can proceed.

- Documented `CurrencyConversion` as a synchronous cached-data callback,
  making the lock and re-entry constraint explicit for users implementing the
  trait.

- Moved terminal `UnverifiableAmount` and `ExcessiveAmount` validation after
  pending-payment lookup so rejection abandons the payment it consumes instead
  of leaving stale state behind.

- Kept insufficient amount handling narrower where corrected data or existing
  flow behavior can still resolve the payment.

### Making payable amounts the shared boundary

- Routed `InvoiceBuilder::amount_msats()` through
  `payable_amount_msats()` so explicit invoice response amounts share the same
  converted minimum and maximum-value checks.

- Preserved reference-rate quoting for requests without explicit amounts while
  rejecting converted amounts whose lower bound or floored result reaches zero.

- Loosened the converter lifetime on invoice response builders so callers can
  pass short-lived conversion snapshots.

- Updated Rustdoc and changelog text around inferred payable amounts, public
  amount-error variants, pending-payment downgrade risk, and claim-time
  conversion failure.

### Filling review gaps without flattening the stack

- Added focused coverage for unsupported payer currencies, converted lower
  bounds with quantity and tolerance, exact-boundary invoice responses, and
  static-invoice `InsufficientAmount` cleanup.

- Reworked the invoice-request deserialization fuzzer to derive rates and
  tolerances from fuzz input instead of stopping early at
  `NullCurrencyConversion`.

- Grouped the PR update around currency flow support, underpayment protection,
  payment cleanup, locking documentation, API documentation, and test
  coverage.

- Repaired rebased upstream outbound-payment tests at the commits that changed
  `OfferBuilder::new` and `InvoiceRequest::respond_with_no_std`, keeping the
  stack aligned with the contracts each commit introduces. ([Branch][currency])

## Reflections

The dummy-hop work was mostly about keeping two true statements from being
collapsed into one. The sender needs an aggregate relay policy that is
conservative enough for pathfinding. The forwarding code still applies the
actual policies selected for each dummy hop, with integer rounding at each
step. Tests had been asking one number to stand in for both. Once deterministic
test policies removed the random movement, the remaining failures showed where
the aggregate was useful for safety but not reversible into exact per-hop
accounting.

That made the currency work feel familiar even though the code was different.
Here too, the pressure was placement. A payer that cannot price an offer should
not sign a request. A recipient should not accept an async claim below the
converted minimum. A terminal invoice amount failure should not return an error
while leaving the pending payment alive. The code became stricter where an
amount would otherwise move forward with a meaning no one had verified.

The arithmetic edges were small but important because they protected that same
line. A tolerance that floors the accepted exchange rate to zero, or a
non-zero currency amount that converts into zero msats, breaks the promise
before higher-level validation even has a chance to reason about it. Moving
those checks into the shared payable-amount path kept the invariant plain:
converted payment amounts must be positive, bounded, and checked through the
same code before they are trusted.

The later coverage and rebase work kept the branch from turning into a pile of
correct fixes that are hard to review. Each new test stayed near the commit
whose behavior it proved. The fuzzer reached conversion math instead of only
unsupported-currency handling. The rebase repair landed where the changed API
first required a converter, not as a loose patch at the end. That mattered
because the branch now spans payment construction, invoice response building,
pending state cleanup, docs, fuzzing, and downgrade notes. The stack is easier
to carry when each commit still explains the boundary it changes.
