# Log 381: What the Token Proves

> 11 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  reviewing the open continuation-token stack against the draft recurrence spec
  and narrowing the next implementation step before writing more code.
- Identified two gaps that still made the recurrence path too loose: request
  recurrence tokens were accepting empty values, and follow-up recurring
  invoices for basetime-less offers still depended on reaching back to
  `offer_recurrence_base`.
- Reframed the token around what it has to authenticate, not only what it has
  to carry: the useful recurrence state is the payer key, recurrence counter,
  recurrence start, and recovered basetime, while `offer_id` should stay
  authoritative from the request instead of being duplicated into the token.
- Moved the recurrence ownership line into `flow.rs`, where token verification,
  `token_basetime` recovery, and effective basetime resolution can happen from
  full request context before invoice construction starts.
- Settled the basetime rules for the remaining cases so the invoice side can
  receive one already-resolved `Option<u64>`: explicit offer basetime wins,
  first recurrence without one falls back to `created_at`, and follow-up
  recurrence without one must come through a verified token.

## Reflection

Today helped by making the continuation token answer a stricter question.

Up to this point I was still treating it a little like a way to smuggle one
timestamp through the recurrence flow. That was why the remaining invoice-side
gap kept feeling awkward. Recovering a basetime does matter, but it does not
say enough on its own about whether this recurring request actually belongs to
the stream that claims it.

Once I asked what the token was supposed to prove, the ownership line got much
easier to place. The layer that verifies recurrence context should also decide
whether the request is allowed to continue. That made `flow.rs` the honest
home for the token logic, because it already has the current request, the
offer-side recurrence rules, and the information needed to decide how
basetime should resolve in each case.

That left the day feeling more settled than the diff itself might suggest. The
useful progress was not one more recurrence helper. It was getting the stack
back to a place where the token reads like authenticated recurrence state
instead of extra plumbing the invoice builder is somehow expected to interpret
later.
