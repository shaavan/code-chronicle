# Log 371: Keeping the Distinction

> 27 April 2026

## What I worked on

- [Continued] the recurrence work on the `rust-recurrence` branch by reviewing
  and tightening the top feature commit around what it actually changes.
- Reworked the public recurrence model on `Offer` so the code now preserves the
  real wire-level distinction between optional recurrence, compulsory
  recurrence without a base, and compulsory recurrence with a base.
- Fixed offer parsing and serialization around that model, including accepting
  valid compulsory-without-base offers and rejecting invalid recurrence
  combinations with `InvalidRecurrence`.
- Rewrote the feature commit message so it describes the real scope of the
  change instead of reading like a smaller field-addition patch.

## Reflection

Today was mostly review work, but it changed the shape of the recurrence
surface in a way that felt worth treating as its own day.

The original implementation was close enough to pass as progress, which is what
made it dangerous. It was deriving recurrence type from the presence of
`recurrence_base`, and that meant the code had quietly collapsed a real wire
distinction into a convenient internal guess. Once I checked it carefully
against the draft BOLT 12 rules, that shortcut stopped feeling harmless. The
model was losing information that the protocol actually carries.

Fixing it was not just a parser adjustment. It needed a public shape that could
hold the three real cases without inference doing hidden work underneath:
optional recurrence, compulsory recurrence with no base, and compulsory
recurrence with a base. Once that became explicit in the type model, the rest
of the logic got sharper too. The parser could accept the combinations the wire
format really allows, reject the ones it does not, and stop misclassifying
recurrence mistakes as generic metadata problems.

The commit message rewrite ended up belonging to the same day for the same
reason. The old message was underselling the change in exactly the way the old
model was underspecifying it. By the end, both the code and the description
were saying the same thing plainly: this patch is not just adding recurrence
fields, it is deciding how much of the protocol's distinction the public API is
willing to keep.
