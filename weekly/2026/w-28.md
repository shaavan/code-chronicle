# Weekly Log 28: State Already Waiting

> 16 July 2026 – 18 July 2026

## Overview

This week moved between the dummy-hop follow-up and the async static invoice
refresh branch, but both threads kept asking the same practical question: what
state is already carrying a promise, even before the rest of the system notices
it?

The dummy-hop work narrowed review feedback into smaller shapes. Fee buckets
were made broad enough to stop preserving unusual exact values. The new skimmed
fee event field moved to an odd TLV so older readers can skip it, and deferred
onion persistence lost a nested lookup table once the real invariant became
positional alignment with pending HTLCs. The branch felt better when each
stored value carried only the structure needed to recover it.

The static-invoice work then followed stale cached invoices back to the channel
events that create them. Channel close finalization now owns the direct refresh
it can make necessary. Peer `ChannelUpdate`s still stay batched, because inline
refreshes would turn forwarding churn into repeated server updates. The forced
refresh selector also stopped treating `Ready` offers as safe. If a cache entry
already contains a server-side static invoice built from old channel state, it
can be stale before the application ever hands it out.

## Highlights

### Paring dummy-hop data back to its real invariants

- Tightened dummy-hop fee buckets so base and proportional fees describe common
  values instead of keeping rare exact amounts alive through tests.

- Moved `dummy_skimmed_fee_msat` from even TLV `12` to odd TLV `11`, preserving
  downgrade behavior for older versions that do not know the new event field.

- Reworked deferred onion decoding persistence from a nested map into a flat
  list whose values line up by position with TLV 14's pending HTLCs.

- Restored public `DummyTlvs` fields after the constructor turned out not to
  protect a stronger invariant than direct construction.

- Resolved the `PaymentClaimable` rebase conflict and corrected the remaining
  payer-proof offer test around the conservative dummy-hop aggregation gap.
  ([Branch][dummy-follow])

### Putting refresh ownership at the channel boundary

- Moved channel-close static invoice refreshes into `finish_close_channel`,
  where the close has been finalized and the stale channel information can be
  acted on directly.

- Kept peer forwarding updates on the pending-refresh path so repeated
  `ChannelUpdate`s can be coalesced before server-side static invoices are
  rebuilt.

- Revisited splice-triggered refreshes and left them out of the forced path for
  the current amountless async receive cache model.

- Separated the current LDK async receive cache from the broader
  `StaticInvoice` abstraction while checking the currency branch's
  `InsufficientAmount` path for externally built currency-denominated offers.

### Refreshing invoices already waiting in cache

- Changed forced refresh selection to include every cached offer with a
  server-side static invoice, including entries still in the `Ready` state.

- Updated refresh documentation so stale invoices are tied to channel state
  changing underneath stored payment details, not to offer age or application
  use.

- Replaced the local `flow.rs` macro for `ServeStaticInvoice` construction with
  a shared closure.

- Fixed async payment tests to allow unrelated `OfferPathsRequest` setup
  messages while still rejecting any inline `ServeStaticInvoice` update before
  the timer tick. ([Branch][static-update])

## Reflections

The dummy-hop review work was useful because it kept removing structure that
looked reasonable until the invariant underneath was named. Narrow fee buckets
made tests more exact than the privacy shape required. An even TLV made the new
field visible to old readers in the wrong way. A nested persistence map named
the lookup by channel and HTLC even though the later value only needed to stay
in order with the pending HTLC list. Each fix made the branch less clever and
more direct.

That same pressure showed up in the static-invoice refresh work, but with
cached payment details instead of skimmed fees. A closed channel is not just
another state change to batch for later; once `finish_close_channel` finalizes
it, any server-side static invoice that depended on that channel can already be
wrong. Peer updates still have a different rhythm. They may arrive as churn,
and the invoice cache should mark itself stale without sending every
intermediate version to the server.

The `Ready` cache state was the part that made the boundary feel less
comfortable. It was tempting to treat unused entries as harmless, but a ready
offer can already contain a static invoice built from old channel state. The
application boundary sees that entry as available. That means the refresh code
cannot wait for use to prove staleness. If the stored invoice depends on channel
state, it belongs in the forced refresh set.

The test fixes mattered because they kept that distinction narrow. Queue
emptiness was too broad an assertion once setup paths could leave unrelated
`OfferPathsRequest`s behind. The invariant was never that no messages existed;
it was that peer updates should not send a `ServeStaticInvoice` update inline,
and the timer should send the coalesced update later. Letting the helper permit
only unrelated setup messages kept the test strict without making it depend on
incidental silence.

By the end of the week, both branches had moved toward plainer ownership.
Dummy-hop data now keeps the order and compatibility it actually needs.
Static-invoice refreshes now follow the channel-state dependency even when the
affected offer is still waiting in cache. The remaining larger question is
amountful offline receive, which does not fit cleanly into the current reusable
amountless cache and should probably enter through a larger API shape rather
than another hidden condition in refresh logic.
