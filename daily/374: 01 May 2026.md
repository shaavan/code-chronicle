# Log 374: Carrying It Through

> 01 May 2026

## What I worked on

- [Continued] the recurrence work on the `rust-recurrence` branch by pushing
  the `InvoiceRequest` fixes past builder semantics and into the request data
  model itself.
- Grouped `counter`, `start`, and `cancel` under a new
  `InvoiceRequestRecurrence` struct and threaded that shape through
  `InvoiceRequestContentsWithoutPayerSigningPubkey` and `InvoiceRequestFields`.
- Kept wire compatibility intact by leaving the existing internal
  `counter` TLV in place and adding new optional TLVs for `start` and `cancel`
  instead of rewriting old encoding.
- Updated the affected call sites and fixtures to the grouped recurrence shape,

  formatter behavior.
- Verified the stack again once the request-side recurrence changes settled.

## Reflection

Yesterday was mostly about making the builder and parser agree. Today was about
whether that agreement was still dropping information on the floor.

Once I followed the recurrence path all the way through `InvoiceRequest`, it
was clear the surface was still carrying the old split too far. `counter` had
a place to live, but `start` and `cancel` were not moving through the same
path cleanly, which meant the request model was acting more complete than it
really was. Grouping those fields under one recurrence shape fixed more than
the API spelling. It made the request side admit that these values belong to
one piece of state and should travel together.

The compatibility part took most of the care. Reusing the old `counter` TLV
for a broader meaning would have made the internal model look tidier while
quietly changing wire expectations underneath it. Leaving that piece alone and
adding separate optional TLVs for the missing fields felt less elegant at
first, but more honest. The code now carries the fuller recurrence shape
without pretending old encoding decisions never happened.

What I liked about the end state was that it narrowed the kind of mismatch I
still have to watch for. The builder checks are stricter, the request model is
less fragmented, and the plumbing no longer loses part of the recurrence
context on the way downstream. What remains is also more sharply bounded.
`recurrence_paywindow` is still waiting on timing and prior-state context that
this layer does not have yet, but the missing piece now looks like one real
unfinished boundary instead of several half-connected ones.
