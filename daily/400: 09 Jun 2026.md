# Log 400: Bounds That Hold
> 09 June 2026

## What I worked on

- Continued the currency work on the `currency-108` branch by tightening the
  exchange-rate model after the fiat-denominated offer flow was already in
  place.
- Reworked the rate primitives so common exchange rates could be represented
  directly, while fractional rates moved behind a more explicit construction
  path.
- Replaced percentage-based tolerances with basis-point tolerances and moved
  accepted-bound validation earlier, rejecting impossible lower bounds and
  unrepresentable upper bounds at construction time.
- Updated offer-amount conversion and the surrounding test utilities to work
  through validated bounds and accessors instead of reaching into raw
  `ExchangeRate` fields.
- Clarified `ChannelManager::pay_for_offer` docs so it is explicit that a
  fiat-denominated `amount_msats` is not enforced locally when building an
  `InvoiceRequest`; the payee still decides whether the offered amount is
  sufficient when issuing the invoice.

## Reflection

This day came after the more visible currency work was already done, which made
it easier to see what still felt loose underneath it.

The end-to-end flow existed. Offers could carry fiat-denominated amounts,
invoice requests could preserve them, and builders could convert them. But once
that path was in place, the exchange-rate model still asked later code to be
too careful around values that should already have been proved safe earlier.
Too much of the surrounding logic still depended on callers handling raw fields,
derived bounds, and overflow-shaped edge cases that belonged closer to the
construction point.

What I wanted by the end was not a bigger feature. I wanted a model that made
fewer ambiguous states available in the first place. Moving tolerances into
basis points helped make the acceptance policy more exact. Rejecting bad bounds
up front mattered even more, because it meant later conversion paths could stop
acting like each use site had to defend itself from the same category of bad
input again.

That gave the work a different kind of satisfaction than the earlier currency
day. The important part was not that more behavior was unlocked. It was that
the existing behavior started resting on something narrower and firmer, with
less room for accidental misuse hiding behind a technically functional API.
