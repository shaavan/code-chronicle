# Log 406: What the Helpers Carry

> 18 June 2026

## What I worked on

- Continued the dummy-hop fee work on the `dummy-follow-26` branch by
  reworking the shared blinded-payment test helpers so they can derive
  dummy-hop fee effects from runtime events instead of local hardcoded
  expectations.
- Threaded `dummy_skimmed_fees_msat` from `PaymentClaimed` into downstream
  helper expectations, which let sent-side fee assertions pick up dummy-hop
  effects without asking each test to precompute them by hand.
- Added helper routines for reconstructing dummy-hop receive-side fee behavior
  and total extra fees, then tightened their comments so the amount flow still
  reads honestly once integer rounding gets involved.
- Split that work as test-framework infrastructure rather than folding it into
  the earlier runtime changes, because the main result today was a better way
  to prove the feature than another change to the feature itself.

## Reflection

After the receive-side accounting and relay-parameter work, the next thing that
started to feel brittle was not the feature itself but the test scaffolding
around it.

The old helpers still assumed they could understand blinded payments from the
outside through exact expected fees and exact path amounts. That had held well
enough before, but dummy hops changed the shape of the problem. Some of the fee
movement now comes back through runtime events, some of it has to be
reconstructed from helper logic, and once rounding enters the path the numbers
stop lining up in the neat inverse way the older assertions quietly depended
on.

What mattered today was moving that understanding into the shared framework
instead of leaving it scattered across call sites. Once `PaymentClaimed` became
the place where dummy skim reentered the helpers, the rest of the expectation
chain became easier to trust. The sent-side fee checks no longer needed each
test to carry its own little private model of dummy-hop behavior, and the
receive-side helpers could say more plainly what they were reconstructing and
why.

I liked this one because it made the work feel less fragile. The runtime path
had already started holding together. Now the tests were beginning to hold in
the same way.
