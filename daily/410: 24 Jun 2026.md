# Log 410: Trust the Conversion

> 24 June 2026

## What I worked on

- [Continued] the currency-conversion PR by sorting the latest review feedback
  into correctness issues, coverage gaps, and lower-value follow-ups.
- Changed the recipient-side currency-payment path to fail closed when the
  conversion cannot be verified, instead of accepting a payment whose converted
  amount the recipient cannot trust.
- Kept `ExchangeRateBound` in place and added focused coverage for fractional
  basis-point ranges, absolute tolerances, invalid lower bounds, and overflow
  behavior.
- Split the offer-builder changes into a more reviewable order, then expanded
  validation tests for supported currencies, unsupported currencies,
  zero-floor conversions, and invalid converted amounts.
- Added async/static invoice coverage showing that a currency-denominated async
  offer still pays the converted amount through a static invoice, while keeping
  the final amount distinct from the outbound HTLC amount with routing fees.

## Reflection

Today was mostly review work, but it did not feel like answering comments one
by one. The currency branch had reached the point where the hard part was
deciding which feedback should change the shape of the feature and which parts
would only make the stack busier.

The recipient-side failure case was the easiest one to take seriously. If a
node receives a payment for a currency-denominated offer and cannot verify the
conversion, accepting that payment leaves the recipient trusting an amount it
cannot price. Making that path fail closed gave the behavior a sharper edge,
and the documentation now explains the constraint where it matters.

The exchange-rate work had a different texture. It was tempting to keep pulling
on the `ExchangeRateBound` structure because the review had put attention
there, but the better move was to make the existing boundary prove itself.
Tests for fractional basis-point ranges, absolute tolerances, bad lower bounds,
and overflow did more for the branch than another API reshuffle would have.

The async/static invoice test was the piece that made the day feel complete.
That path hides the currency amount inside a helper, so the important question
was whether the payer still derives the actual payment amount from the original
request and configured converter. The test now follows that all the way through
and also catches the small accounting detail that the outbound HTLC includes
routing fees above the final converted amount.

By the end, the branch felt less like a collection of review replies and more
like a tighter argument. The behavior fails in the right direction, the public
builder surface carries the explanation callers need, and the coverage reaches
the paths where currency conversion would be easiest to accidentally lose.
