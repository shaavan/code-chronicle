# Weekly Log 23: Where State Belongs

> 03 June 2026 – 07 June 2026

## Overview

This week kept circling one question across very different pieces of work:
where does the system actually have enough context to say something
defensible, and what breaks once that context is forced to live somewhere
else.

That showed up first in the currency branch, where the accepted conversion
range stopped being a half-hidden expansion of derived numbers and became one
quoted rate plus explicit tolerances. It showed up again in the recurrence
work, where several days of implementation pressure finally turned into a spec
writeup that could name exactly which parts of the current model ask too much
guesswork from the implementer. Then the same pressure reappeared in two
different architectural places: dummy-hop fee accounting, where receive-side
state was getting lost between queued onion peels, and logging review, where
the patch was speaking from block connection when the real events lived in the
sync layer. By the end of the week, the useful progress was less about pushing
one branch further and more about getting stricter about where meaning is
allowed to originate.

## Highlights

### Making currency acceptance visible in the model

- Reworked the `currency-107` branch around one quoted exchange rate with
  explicit lower and upper tolerances instead of carrying around a pre-expanded
  exchange-rate range that had already lost the policy that created it.
  ([Branch][Currency])

- Moved accepted-bound calculation into one place, so callers no longer have
  to pass around derived minimum and maximum rates without owning how those
  numbers were formed.

- Tightened converted-amount validation near `MAX_VALUE_MSAT` by clamping an
  oversized upper bound instead of discarding an otherwise payable range, then
  carried that model through builders, `ChannelManager`, tests, fuzz helpers,
  and `c_bindings`.

### Turning recurrence pressure into a reviewable design

- Finished the recurrence-spec writeup after several days of design rework,
  turning implementation discomfort into a concrete account of what the current
  recurrence model makes too hard to reason about or too easy to implement
  loosely.

- Named the privacy and reliability failures directly, including the problem
  with keeping `payer_signing_pubkey` stable across recurring payments and the
  broader cost of recurrence forms that still depend on too much local
  interpretation.

- Consolidated the replacement ideas into one coherent design that is ready to
  put in front of other contributors, instead of keeping them as scattered
  fixes living partly in notes and partly in prototype code.

### Finding the right boundary for carried context

- Traced the receive-side dummy-hop flow until the main problem stopped looking
  like fee arithmetic and started looking like lost context between queued
  onion peels, because each dummy hop re-enters ordinary forwarding machinery
  without a stable place to keep prior accounting. ([Branch][Dummy])

- Ruled out recursive peeling because it would preserve local state by making
  paths with different dummy-hop counts behave differently, then developed a
  cleaner direction based on receiver-generated, HMAC-authenticated
  continuation state.

- Reviewed PR [#4659] against issue [#2348] from the same angle: not "where
  can more logs be added," but where LDK first has enough context to say that
  chain activity became relevant. That led the [review] toward the sync layer,
  along with missing reorg-side unconfirmations, filter-registration
  visibility, and better batch summaries.

## Reflections

The week felt strongest whenever I stopped treating misplaced context as a
small implementation inconvenience.

The currency work made that visible in a relatively contained way. A pre-built
range could move through the stack, but it blurred the more important fact
that the application is really accepting one quote within some tolerance. Once
the model carried that truth directly, a lot of the surrounding code got more
honest with it. The point was not elegance. The point was that downstream code
no longer had to pretend it understood numbers whose origin had already been
erased.

The recurrence writeup did the same thing at document level. Earlier in the
week, the pressure still showed up mostly as code that felt harder to trust
than it should. Writing the design down forced sharper claims: which parts of
the current spec require too much interpretation, which parts would drift
across implementations, which parts keep more continuity than repeated
payments should. Once those failures sat in one writeup instead of in a
moving prototype, the work finally looked ready for criticism from other
people rather than only from my own implementation notes.

The dummy-hop work and logging review pushed that lesson into architecture.
In both cases, the bug was not that information existed nowhere. It was that
the chosen edge of the system was too early or too forgetful to speak for it.
Queued dummy-hop processing dropped fee state between peels. Block-connection
logging could see transactions arrive, but not as naturally tell the fuller
story of relevance, unconfirmation, and filter movement. Once I followed those
problems back to the place that actually owns the missing context, the next
steps got narrower. That is what this week clarified: several of these threads
are now less about adding behavior and more about refusing to let the wrong
layer guess on behalf of the right one.
