# Log 411: After the Locks Drop

> 25 June 2026

## What I worked on

- [Started] the static-invoice refresh branch by turning the async-receive
  staleness issue into an event-driven refresh path for channel changes.
- Added cache-level selection for forced refreshes, covering used offers that
  may already be published and pending offers that already fit the retry model.
- Threaded the forced-refresh path through `OffersMessageFlow` and
  `ChannelManager` while keeping static invoice construction on the existing
  timer-refresh builder.
- Marked refreshes from channel ready/open, channel close, and local fee-update
  paths without rebuilding invoices while channel or peer locks are still held.
- Added focused coverage showing that opening a new channel to the static
  invoice server immediately emits a replacement `ServeStaticInvoice` for the
  same used-offer slot.

## Reflection

Yesterday had left the async-payment work at the boundary where static invoice
data stops being truly static. Today was about making that boundary usable in
code. The existing receive path already knew how to refresh invoices over time,
but timer freshness was not enough for data built from live channel state. A
new channel, a close, or a fee update can change the blinded paths before the
two-hour threshold has anything to say.

The first useful constraint was deciding what should be refreshed at all. Used
offers were the easy case because the application may already have handed them
out. Pending offers also made sense because another invoice send follows the
same shape as their existing retry path. Ready offers were more awkward. They
are cached, but not yet exposed, so forcing them through the server-side refresh
path would add churn where unused-offer rotation already has the better job.

Once that line was drawn, the implementation mostly became a lock-ordering
problem. Rebuilding a static invoice needs a fresh view of usable channels and
can walk back into routing and message construction. That is exactly the kind
of work I did not want hidden inside channel-ready, close, or fee-update
handling while other state is still borrowed or locked. The pending flag made
the behavior feel less clever and more solid: channel paths only record that
the invoice set is stale, then the refresh happens after the surrounding work
has let go.

I also liked that the refresh did not grow its own invoice builder. The cache
selector changed, but the `ServeStaticInvoice` construction stayed on the same
path as timer refresh. That gives the new behavior a smaller review surface.
There is only one place that decides how to build the invoice, and the event
trigger only decides when an already-published or retrying offer needs that
builder again.

The test helped pin the shape down. It does not wait for time to pass or try to
prove every channel transition at once. It takes a used offer, opens another
channel to the static-invoice server, and checks that the replacement invoice
arrives immediately for the same slot. That is the behavior the issue needed:
not a more eager timer, but a refresh point tied to the channel state that made
the old invoice stale.

By the end, the stack had the right kind of narrowness. Cache selection,
plumbing, event trigger, and coverage each have their own commit, and the
history cleanup made that order easier to read. The part that matters most is
where the work does not happen. The invoice only gets rebuilt after the channel
transition has stepped out of the lock-heavy part of the code, which keeps the
static-invoice path live without making it invasive.
