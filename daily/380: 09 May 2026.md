# Log 380: What the Context Holds

> 09 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  moving the missing basetime state for recurring offer payments into
  payer-side offer-payment context instead of trying to recover it during
  invoice parsing.
- Added `expected_recurrence_basetime: Option<u64>` through
  `OptionalOfferPaymentParams`, `OffersContext::OutboundPaymentForOffer`,
  retry-preserved payment state, and the blinded reply-path context used while
  constructing the `InvoiceRequest`.
- Reworked recurring invoice verification to use that carried context
  directly: follow-up recurring invoices must match the expected basetime
  exactly, while the first recurring invoice for a basetime-less offer is the
  only path allowed through without a prior expectation.
- Tightened the surrounding invariants so impossible states now fail early,
  especially supplying an expected basetime on non-recurring or compatibility
  one-off requests, or reaching a follow-up recurring verification path without
  one.
- Verified the change with a full local `lightning` test run and added focused
  follow-up coverage, including a fix for a debug-only path that had been
  tripping `debug_assert!` instead of returning
  `Bolt12SemanticError::InvalidMetadata`.

## Reflection

Today got better once I stopped asking invoice parsing to discover something it
could never honestly know yet.

The awkward case was the first recurring invoice for an offer that does not
publish a recurrence basetime. The recurrence stream still needs one stable
`invoice_recurrence_basetime`, but when the payer builds the first
`InvoiceRequest`, that value has not been established yet. I had been circling
around that gap from the invoice side, which made the code feel more derived
than it really was. Once I moved the question back to outbound payment context,
the shape settled down. The payer is already the place that knows whether this
payment is the first one in the stream or a follow-up that should already be
anchored.

That changed the verification story in a useful way. Instead of teaching the
invoice layer more recurrence guesswork, the returned invoice now gets checked
against context the payer deliberately carried into the request. When there is
an expected basetime, the invoice has to match it. When there is not, that
absence means one very narrow thing: this is the first recurring invoice in a
basetime-less stream, and only that case gets through.

I liked how much simpler the failure paths became once the ownership was right.
The impossible combinations stopped looking like loose optional-field bugs and
started reading as actual recurrence invariant breaks. Even the test-found
`debug_assert!` issue fit that pattern. Fixing it made the edge case behave the
same way the rest of the path was already trying to behave: reject bad input
plainly instead of only catching it in debug builds.
