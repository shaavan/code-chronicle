# Log 382: Where the Flow Decides

> 13 May 2026

## What I worked on

- [Continued] the BOLT 12 recurrence work on the `rust-recurrence` branch by
  moving recurrence-token verification and basetime resolution out of the
  invoice builder and into the offer response flow that already has the full
  request context.
- Simplified the token format in `signer.rs` by removing `offer_id` from the
  recurrence token and cutting the older layout helpers, leaving the token to
  carry only encrypted basetime, nonce, and MAC.
- Threaded an explicit `Option<u64>` recurrence basetime through the
  `respond_*` paths into invoice construction, then removed the older
  builder-side fallback logic that had still been trying to infer recurrence
  state from partial inputs.
- Extended the recurring invoice path so the payee now emits the next
  `invoice_recurrence_token` from resolved state, fresh nonce material, and
  the existing expanded key, while keeping that continuation token gated off
  for cancellations.
- Updated the invoice tests around the new ownership line, including coverage
  for auto-generated continuation tokens, follow-up recurrence with resolved
  basetime, and the cancelled path that should not mint a next token, then
  verified the stack locally with the default toolchain after the older Cargo
  1.75.0 path ran into the `Cargo.lock` version mismatch.

## Reflection

Today got easier once the recurrence decision finally lived in the same place
that already had enough context to make it.

The builder-side fallback had been bothering me because it was still pretending
invoice construction could recover recurrence meaning from partial state after
the important questions had already gone by. Once token verification and
basetime resolution moved into `flow.rs`, the stack stopped feeling like it was
passing around hints and started behaving like one path that actually knows why
it is allowed to continue.

That made the continuation token work feel more real too. Emitting the next
token from resolved recurrence state is a much better fit than treating the
current token as something the invoice layer merely consumes and forgets. The
stream now carries itself forward from the same checked context that admitted
the current request in the first place.

I liked how much code fell away with that move. Removing `offer_id` from the
token and deleting the builder fallback both came from the same correction:
stop duplicating information or decisions in layers that only know part of the
story. By the end of the day, recurrence handling felt less split across the
stack, and the invoice path finally looked like it was continuing one coherent
flow instead of patching around the edges of one.
