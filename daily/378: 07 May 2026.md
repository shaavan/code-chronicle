  pubkey.

## Reflection

Today helped by turning a feature addition back into an ownership question.

The explicit payer pubkey change looked reasonable at first because recurring
invoice requests really do need a stable `invreq_payer_id`. But once I looked
closely at how the builder was meant to finish, the shape stopped holding. The
same surface was trying to support both the privacy-first derived-key flow and
an explicit-pubkey flow that did not actually own the signing key needed to
call `build_and_sign`. That made the API look more capable than it was safe to
be.

Going back through the earlier LDK history helped here. The older removal of
user-supplied payer signing pubkeys already carried the warning I needed:
letting callers override identity too casually creates privacy and ownership
problems at the same time. The better answer was not to force the explicit key
path into the default builder. It was to separate it and make the cost visible.
If the caller wants to supply the payer pubkey directly, they also need to own
the final signing step.

That left the branch in a better state than a simple cherry-pick would have.
The recurring-request work still moves forward, but the public surface now says
more plainly which path can safely sign for you and which one only prepares the
request for a caller who is taking that responsibility on themselves.
