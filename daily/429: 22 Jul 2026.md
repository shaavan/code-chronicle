# Log 429: What Recurrence Remembers

> 22 July 2026

## What I worked on

- Studied the [recurrence spec PR] and focused on the places where long-lived
  recurrence changes the shape of the protocol.
- Questioned whether `invoice_recurrence_next_state` and
  `invreq_recurrence_prev_state` are enough for a stateless payee when invoice
  issuance and invoice settlement can diverge.
- Explored whether recurrence invoices need a way to refresh blinded request
  paths so sessions can outlive the original `offer_paths`.
- Revisited the cost of supporting both fixed and payer-relative recurrence
  anchors, especially around `invoice_recurrence_basetime`.
- Discussed whether compulsory recurrence belongs in the protocol or whether
  subscription-only behavior should remain merchant policy.

## Reflection

Today was mostly a reading and discussion day, but it did not feel loose. The
recurrence spec has enough small fields that it is easy to read each one in
isolation and miss the longer pressure. Recurrence is not only an invoice being
asked for again. It is a relationship that has to stay meaningful across
payment windows, route changes, and decisions about who remembers what.

The first question was around statelessness. `next_state` and `prev_state`
help the payee avoid storing recurrence-specific progress, but they are tied
to invoice issuance. That leaves an awkward edge when payment windows overlap:
a payer can receive continuation state for invoice `n` without necessarily
paying it, then use that state to request invoice `n+1`. If the payee is meant
to validate recurrence completely statelessly, issuance proof may not be
enough. The next request may need to prove settlement too, maybe by carrying
the previous payment preimage alongside authenticated recurrence state.

That made the state itself feel more loaded. It may have to commit to the
recurrence session, payer id, counter, optional start, previous payment hash,
amount or quantity details, and cancellation or validity rules. At that point
the state is no longer just a continuation token. It becomes the thing that
lets the payee reconstruct the part of the relationship it deliberately chose
not to store.

The path question had a similar shape. A recurrence offer with
`offer_issuer_id` has a stable target for future invoice requests. A recurrence
offer that began through blinded `offer_paths` depends on paths that can go
stale while the subscription itself is still alive. Channel closures, peer
changes, and different introduction nodes can all break the original message
route. If the protocol only knows about those original paths, a valid
recurrence session can die for a reason that has little to do with the
recurrence state.

Fresh blinded request paths in later recurrence invoices would solve that in a
different layer from payment. `invoice_paths` are for paying the current
invoice. Request paths would be for keeping the next invoice request
deliverable. That distinction made the long-lived version of recurrence feel
more concrete: reachability is part of the subscription machinery, not a
detail the first offer can always settle forever.

The anchoring discussion was more about whether the current flexibility is
worth the branching it creates. Supporting both a fixed
`offer_recurrence_base.basetime` and a payer-relative basetime derived from the
first invoice means every step has to keep track of which model is active.
`invreq_recurrence_start`, `invoice_recurrence_basetime`, and payee-side memory
all change shape depending on that choice. A single offer basetime would make
the protocol smaller, though it would also give up one static offer creating
independently anchored subscriptions for many payers.

The optional-versus-compulsory question was the smallest one, but it still
pressed on the same boundary between protocol and policy. If recurrence is
always optional, the protocol only describes how recurrence works and the
merchant decides whether one-time payments are acceptable. The cost is an
extra rejection path for wallets that try a one-time request against a
subscription-only merchant. That may be acceptable, or it may be too much
friction to hide behind policy.

No code moved today, but the spec felt less abstract by the end. The difficult
parts were not in any one TLV. They were in the lifetime of the relationship:
what proves payment, what keeps requests reachable, what anchors time, and
what the protocol should refuse to encode as merchant policy.
