# Log 403: What Survives the Queue

> 12 June 2026

## What I worked on

- Continued the dummy-hop fee-accounting work by moving from design direction
  into a full receiver-side implementation that still keeps dummy-hop use
  hidden from senders.
- Added recursive dummy-skim accumulation in the local blinded-onion peel path
  so each dummy layer contributes its fee delta before the real receive
  payload is reached.
- Extended the same accounting through `ChannelManager`'s deferred dummy-hop
  flow by storing the accumulated skim alongside pending
  `update_add_htlc`s instead of changing the existing one-peel-per-iteration
  behavior.
- Carried the accumulated value into receive-side payment state so it can be
  aggregated across MPP parts and surfaced as `dummy_skimmed_fees_msat` in
  both `PaymentClaimable` and `PaymentClaimed`.
- Updated the relevant serialization paths so the new accounting survives
  replay and restart, then finished with formatting and compile-oriented
  checks.

## Reflection

Yesterday's design work was mostly about where this number should live. Today
was about proving that answer could survive the actual receive flow.

The awkward part of this problem is that dummy-hop processing is built to shed
local context as it moves forward. That is part of why the privacy story holds
up in the first place. A dummy layer gets peeled, the next onion message gets
queued, and the system deliberately stops treating it as something special. So
the work only held if I could preserve the fee accounting without undoing that
shape or teaching the sender anything new about the path it had just used.

The deferred `ChannelManager` path made that pressure real. Anything that
depended on stack-local accumulation worked only until the message reentered
the queue. Once I accepted that, the implementation became much more concrete:
the accounting had to move as local receiver state through each peel and then
stay attached to the queued work long enough to reappear at the final payment
event. By the end, the part I liked was not only that the number showed up in
`PaymentClaimable` and `PaymentClaimed`. It was that the receive path still
looked like itself.
