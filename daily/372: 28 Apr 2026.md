# Log 372: Letting the Calendar Speak

> 28 April 2026

## What I worked on

- [Continued] the recurrence work on the `rust-recurrence` branch by replacing
  the old fixed-duration month handling with calendar-based period-start logic.
- Reworked the surrounding helper and doc layout so the public API docs,
  internal comments, and spec notes stop bleeding into each other.
- Checked the implementation with compile-oriented verification and the full
  `cargo test -p lightning` suite, then added targeted recurrence coverage in
  `offers/offer.rs`.
- Found and fixed a real serialization bug while writing those tests:
  `RecurrencePaywindow` was using the wrong integer encoding and was not
  round-tripping correctly.
- Finished the stack with a cleaned-up `[feat]` commit for the time semantics
  work and a separate `[test]` commit for the focused recurrence coverage.

## Reflection

Today started as a semantics pass, but it quickly became a question of whether
the recurrence code was willing to mean the same thing as the draft.

The old month handling was treating a month like thirty days, which is a fine
placeholder right up until the protocol stops being abstract. Once I worked
through the recurrence rules carefully, that shortcut stopped feeling
defensible. A recurring payment tied to calendar months has to land where the
calendar says it lands, even when that means clamping January 31st down into
February instead of pretending duration arithmetic can smooth it over. Putting
that behavior into period-start logic made the implementation feel much less
approximate.

The doc cleanup belonged to the same day for a similar reason. The helper block
had started mixing API contract, implementation notes, and spec commentary in a
way that made the code harder to read than it needed to be. Pulling those apart
did not change behavior, but it made the behavior easier to trust. The comments
now say more clearly which parts are stable surface, which parts are internal
machinery, and where I am making an explicit compromise, like leaving leap
seconds out instead of dragging in a lot of extra machinery for a rare edge
case in a draft protocol.

The useful pressure came from the tests. The full crate suite passing was good,
but the focused recurrence coverage immediately found something real:
`RecurrencePaywindow` was not even encoding itself correctly. I liked that kind
of failure. It meant the tests were doing more than decorating the feature
commit after the fact. By the end of the day, the stack felt tighter in a way
I could point to: the time model was closer to the protocol, the tradeoffs were
named plainly, and the tests had already forced one hidden bug into the open.
