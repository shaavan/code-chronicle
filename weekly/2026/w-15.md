# Weekly Log 15: Putting It Back in Place

> 09 April 2026 – 14 April 2026

## Overview

I spent most of this week moving logic back to the part of the flow where it
actually belongs.

The currency branch came first. I rebuilt the amount-handling path around
`MsatsRange`, replayed the branch on top of it, and kept pushing the BOLT12
fiat flow toward a simpler shape. If a fiat request amount is omitted, it stays
omitted until the payee builds the invoice. That turned out to be the point
where a lot of smaller problems started lining up.

By the end of the week I was back in the old recurrence PoC, looking at the
same kind of problem in a different form. Too much recurrence-specific control
flow had settled into `ChannelManager`, so the work became figuring out what
still belongs in `rust-lightning` and what only exists there because the PoC
grew in place.

## Highlights

### Currency conversion

- Reworked the currency-conversion branch around a canonical `MsatsRange`
  abstraction so amount resolution, validation, and tolerance handling use the
  same model. ([Branch][Currency93])

- Split and replayed the branch history so `CurrencyConversion`, range
  resolution, and surrounding invoice logic land in a cleaner order.

- Expanded coverage around unsupported currencies and invalid conversion paths,
  especially around the edges that had stayed fuzzy before.

### BOLT12 fiat flow

- Finished reshaping the fiat-offer flow so omitted request amounts remain
  omitted on the wire and are resolved by the payee when constructing the
  invoice. ([Branch][Currency95])

- Replaced brittle exact comparisons between independently resolved fiat values
  with payer-side validation against the expected payable range.

- Tightened failure handling so bad or unverifiable fiat invoices fail
  immediately and clear pending state instead of lingering until expiry.

### Recurrence

- Closed the remaining currency review issues around validation, safe range
  handling, and fuzz alignment. ([Branch][Currency97])

- Returned to recurrence and narrowed the next phase to generic protocol seams
  needed for a node-owned lifecycle rather than continuing to grow the old PoC
  in `ChannelManager`.

- Re-audited the earlier recurrence branch against `ldk-node`, then restarted
  the work from `upstream/main` so I could reintroduce only the protocol
  surface that still made sense. ([Branch][Recurrence])

## Reflections

The currency work felt stuck for a while because the same problem kept showing
up under different names. Amount resolution was happening too early, so the
request-building path was carrying work that belongs later. Once that moved
back to invoice time, the branch stopped pulling against itself in so many
places. The wire behavior looked right again, the payer-side checks had a
better basis, and the failure path stopped feeling like an afterthought. It
was satisfying to watch several loose pieces settle once the flow was allowed
to behave more like itself.

I carried that into the recurrence work. Looking back at the old branch, it was
hard to ignore how much of it had turned into session management and lifecycle
control. Some of the builder and signing work still feels like protocol
surface. A lot of the rest does not.

That is why restarting from `upstream/main` felt necessary. I did not want to
keep trimming the old PoC from the outside. Starting over made it easier to ask
of each piece whether it still belongs once the recurrence-specific machinery
is gone. By the end of the week, that reset felt less like throwing work away
and more like arriving at firmer ground for what comes next.
