# Weekly Log 17: Reading for the Real Boundary

> 21 April 2026 – 28 April 2026

## Overview

This week kept moving between documents and protocol work, but it did not split
into two different tracks.

I spent the first half reading the quarter back from outside my own working
memory: first through a quarterly report, then through a proposal for the next
year, then through another pass on the OpenSats report. That kept forcing the
same question in slightly different forms. What, exactly, has become solid
enough to claim? By the time I came back to recurrence, that question had
turned into implementation pressure. The work narrowed toward a cleaner
boundary between `rust-lightning` and LDK Node, then tightened further into
whether the public recurrence model and time semantics were actually willing to
keep the distinctions the draft protocol makes.

## Highlights

### Re-reading the quarter from outside

- Wrote the quarterly report and used it to reconstruct which threads had
  actually compounded over the last few months instead of only feeling large
  while I was inside them.

- Rebuilt the next-year proposal around one concrete claim, making BOLT12
  deployable in LDK, and rewrote the supporting sections so async payments,
  onion-message abuse resistance, custom workflows, and recurrence all pointed
  toward that same operational target.

- Reworked the Q8 OpenSats report from the perspective of an external reader,
  cutting repeated claims, separating older groundwork from this quarter's
  actual movement, and rewriting the recurrence section around the now-clearer
  `rust-lightning` versus LDK Node split.

### Narrowing the recurrence boundary

- Worked through how inbound recurring invoices should be verified without
  requiring mid-flow session lookups from LDK Node during invoice handling.
  ([Continued][Issue4380])

- Settled on a cleaner boundary where LDK Node provides recurrence-session
  verification material when creating the invoice request, allowing
  `rust-lightning` to carry that context through the blinded reply path and
  verify the returning invoice locally.

- Used that design pass to reject the shape of the earlier payer-side PoC as a
  production template and restated the ownership split more plainly: wire
  fields, APIs, and protocol checks in LDK; session state, expiry, updates,
  and policy in LDK Node.

### Making the protocol keep its distinctions

- Reworked the public recurrence model on `Offer` so it preserves the real
  wire-level cases instead of inferring them indirectly from
  `recurrence_base`. ([Branch][Recurrence])

- Fixed parsing and serialization around that model so valid compulsory
  recurrence without a base is accepted, invalid combinations fail with
  `InvalidRecurrence`, and the feature commit now describes the real scope of
  the change.

- Replaced the old fixed-duration month handling with calendar-based
  period-start logic, cleaned up the surrounding API and spec commentary, and
  added focused recurrence coverage that exposed and fixed a real
  `RecurrencePaywindow` serialization bug.

## Reflections

What held the week together was not recurrence by itself. It was the repeated
act of reading the work from farther away.

The report and proposal days kept removing the convenience of proximity. They
made it harder to count motion as position, or to let a broad ambition stand in
for a narrower result that can actually be defended. That pressure turned out
to be useful when I came back to the code. The recurrence design question on
23 April only became simple once I stopped letting the old PoC define the
shape of the solution. If LDK needs to ask LDK Node halfway through invoice
handling whether a recurrence invoice is valid, then the split still is not
clean enough. Moving the verification material earlier made the boundary feel
much more boring in the right way.

The implementation work at the end of the week pushed on the same weakness from
another angle. The earlier recurrence model was close enough to look finished,
but it was quietly throwing away protocol information. The old month logic had
the same problem. It was smoothing over a real calendar rule with a convenient
duration shortcut. Fixing both made the branch feel less like a feature stack I
was trying to preserve and more like a protocol surface I was willing to be
precise about.

By the end of the week, the useful standard felt sharper: not whether a branch
can be explained into seeming reasonable, but whether the boundary and the
model still hold when read without context doing half the work.
